Pi Agent 是什么?

一个极简主义的终端 Coding Agent -- 不预设工作流,核心极小,一切可扩展。

✦
终端原生
TUI 界面,纯键盘驱动,无 Electron 包袱
✦
4 个内置工具
read / bash / edit / write -- 足够完成编码循环
✦
TypeScript 扩展体系
事件驱动、命令注册、工具注册、TUI 面板
✦
树形会话
分支、压缩、合并 -- 比线性对话灵活得多
✦
31+ Provider
Anthropic / OpenAI / DeepSeek / Routinai / 火山引擎 ......

设计哲学对比

Claude Code - 开箱即用,内置 Plan/Agent 模式
Codex CLI - 内置子 Agent、Sandbox、多模型
Pi Agent - 只给你积木,你自己搭
❌ 没有 MCP → 用 CLI 工具 + README
❌ 没有子 Agent → 用 subagent 扩展
❌ 没有 Plan Mode → 用 thinker Agent
❌ 没有权限弹窗 → 用容器隔离

事件生命周期

扩展通过监听事件来介入 Agent 的每个执行环节。以下是完整的事件链。

session_start - 会话开始/恢复时触发
session_tree - 树导航切换时触发
agent_start - Agent 开始处理
before_agent_start → 可注入系统提示
turn_start - 每一轮 LLM 调用的开始
message_start - 消息开始生成
tool_execution_start → tool 开始执行
tool_execution_end → tool 执行完毕
message_end → 可返回修改后的消息
turn_end - 本轮结束
agent_end - Agent 处理完毕

标注 accent 的事件是可以在 handler 中返回数据来修改行为的。

扩展体系:你能改什么

扩展是 .ts 文件,放对应目录自动发现。通过 ExtensionAPI 接口操控运行时。

命令

pi.registerCommand()

注册 /xxx 斜杠命令,绑定 handler

pi.registerCommand("flow", {
  description: "Open flow panel",
  handler: async (args, ctx) => {
    // ctx.ui, ctx.sessionManager, ctx.model 均可访问
  }
});
事件

pi.on()

监听生命周期事件,部分可返回数据修改行为

pi.on("before_agent_start", (e, ctx) => {
  return { systemPrompt: "..." };
});

pi.on("message_end", (e) => {
  return { message: modified };
});
工具

pi.registerTool()

添加 Agent 可调用的新工具(如 subagent)

自动获得 TUI 渲染、参数验证(TypeBox)、流式更新

运行时控制

pi.setActiveTools()

动态切换 Agent 的工具集

还有 getActiveTools()、appendEntry()、ctx.ui.custom()、tui.showOverlay()

/chat ↔ /code 模式切换

灵感:闲聊时不需要 Agent 主动读写文件。
一个命令切换 Agent 的行为能力。

✓
chat 模式
只留 bash 工具,注入极简提示
✓
code 模式
恢复全部默认工具和编码提示
✓
状态持久化
跨 /tree 导航不丢失
✓
状态栏指示
ctx.ui.setStatus() 显示 💬 Chat

部署路径:先在 .pi/extensions/ 项目级开发验证 → 稳定后迁到 ~/.pi/agent/extensions/ 用户级

chat-mode.ts · ~130 行
// 切换前保存完整工具集
savedTools = pi.getActiveTools();

// 进入聊天模式:只留 bash
pi.setActiveTools(["bash"]);

// 持久化到 session 分支
pi.appendEntry<ChatState>("chat-mode",
  { mode: currentMode });

// 注入极简系统提示
pi.on("before_agent_start", async (event) => {
  if (currentMode === "chat") {
    return { systemPrompt: chatPrompt };
  }
});

// 恢复
pi.registerCommand("code", {
  handler: async (_args, ctx) => {
    pi.setActiveTools(savedTools);
    ctx.ui.setStatus("mode", undefined);
  }
});

extension · 3 / 5

routinai-compat:50 行兼容补丁

非标准 Provider 的兼容问题 -- message_end 事件可以返回修改后的消息来修复。

⚡问题:routinai 未持久化 reasoning item ID
⚡pi 把 rs_... ID 嵌入多轮请求 → API 拒绝
✓解法:on("message_end") 剥离 thinkingSignature
✓仅针对 msg.provider === "routinai"

这个 pattern 说明:事件 handler 返回数据可以修改 Agent 行为,形成拦截器链。

routinai-compat.ts · ~50 行
pi.on("message_end", (event, _ctx) => {
  const msg = event.message;
  if (msg.role !== "assistant") return;
  if (msg.provider !== "routinai") return;

  let modified = false;
  const newContent = msg.content.map(block => {
    if (block.type === "thinking"
        && block.thinkingSignature) {
      modified = true;
      const { thinkingSignature, ...rest } = block;
      return rest;
    }
    return block;
  });

  if (modified) {
    return { message: { ...msg, content: newContent } };
  }
});

/flow 工具执行面板

最复杂的扩展。实时显示 Agent 的工具调用流:谁在跑、耗时多少、并行还是串行。

设计目标

✓nonCapturing - 不拦截键盘输入
✓实时 - 工具启动、更新、完成即时刷新
✓并行可视化 - 树形分支线 ├ └ │
✓耗时跟踪 - 运行中的实时计时,完成后的总耗时
✓状态标记 - ◌ 运行中 / ✓ 完成 / ✗ 失败

状态:已可用,多屏滚动渲染仍在优化中。

关键事件

tool_execution_start → 记录工具名、参数、时间
tool_execution_update → 触发重绘
tool_execution_end → 更新状态和耗时
turn_start / agent_start → 切换上下文

并行检测算法

同一 turn 中同时 running 的工具
→ 标记为同一 batch
→ 树形分支线显示

extension · 4 / 5 cont.

/flow 技术难点与解决

非劫持 overlay 技巧
ctx.ui.custom<void>((tui, theme, _kb, done) => {
  // 1. 捕获 tui + theme 引用
  const capturedTui = tui;
  const capturedTheme = theme;

  // 2. 立即弹出 custom controller
  done();

  // 3. 下个 tick 创建 nonCapturing overlay
  setTimeout(() => openPanel(capturedTui, capturedTheme), 0);

  return {
    render: () => [],
    invalidate: () => {},
    handleInput: () => {},  // 空 handler
  };
}, {
  overlay: true,
  overlayOptions: {
    anchor: "top-right",
    width: 1, height: 1,  // 临时面板几乎不可见
  },
});

最大的坑:ctx.ui.custom() 默认劫持键盘。

✗
第一版
custom() 劫持式 → 面板可见但无法输入
⚠
第二版
done() 弹出后创建 overlay → 偶尔崩溃
✓
第三版(稳定)
done() + setTimeout(0) + nonCapturing → 稳定运行

extension · 5 / 5

subagent 多代理协作

基于 SDK 官方示例安装(软链接到 examples/extensions/subagent/)。每个 subagent 启动独立 pi 进程,完全隔离上下文。

Single - 单个委托

subagent({
  agent: "vision",
  task: "分析截图 /path/to/img.png"
})

指定 agent + task,自动匹配模型和工具集

Parallel - 并行

subagent({
  tasks: [
    { agent: "thinker", task: "分析论文A" },
    { agent: "thinker", task: "分析论文B" }
  ],
  agentScope: "user"
})

并发 ≤ 4,结果汇总展示

Chain - 链式

subagent({
  chain: [
    { agent: "thinker",
      task: "分析截图中的报错信息" },
    { agent: "executor",
      task: "根据分析修复代码: {previous}" }
  ]
})

{previous} 自动替换为上一步输出 · 出错自动中断 · 步骤编号 + token 统计

实践后的几点思考

01

事件系统是核心竞争力

其他 Agent 的工具调用是黑盒，Pi 的事件钩子可以拦截、修改、注入消息。30 行 routinai-compat 就是例子——在其他平台根本没法修这个级别的兼容问题。

02

SKILL.md 是被低估的创新

Skills 本质是给 AI 读的操作手册，不是代码。会写 README 就会写 skill。deploy-notes 只用 60 行 Markdown 完成了 cp → commit → push 的完整自动化。

03

扩展开发成本极低

chat-mode 130 行、context 260 行、routinai-compat 50 行——相对于它们解决的问题，TypeScript 类型提示让开发体验很好。

04

树形会话被严重低估

线性对话丢失上下文得重来。树形分支可以探索多个方向再回溯，/compact 压缩后继续。研究型工作流中非常实用。

05

生态虽年轻，但扩展点已够用

Provider 兼容层、事件系统、TUI API 都已可用。欠缺的是社区生态和文档完善度，不过对愿意读源码的开发者不是问题。
