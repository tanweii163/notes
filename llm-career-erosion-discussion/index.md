## 一、原文结构梳理

### 背景

作者是一名深耕金融/支付/记账/对账领域 10 年的后端工程师。他的职业策略是：**成为该领域的专家**，以此在市场上脱颖而出。

---

### 第一根崩塌的支柱：领域特定知识

- 入职一家纯金融公司，被要求用 AI 加速产出
- 自己辛苦积累的支付系统知识（幂等性、对账、escrow 等）被 LLM 轻松复现
- 原以为"调试是 AI 永远做不好的事"——那是他的安全感来源

### 第二根崩塌的支柱：调试和分布式系统

- Claude 4.5→4.6→4.7、GPT 5.5、Opus 4.8 配合 MCP（Sentry、DataDog），**90% 的 bug 被一键解决**
- 包括竞态条件、第三方集成问题、未文档化的 API 边界情况
- 他变成了一个"货架工程师"（off-the-shelf engineer）——所有领域专长和调试直觉都变成了 **"可提示的"（promptable）**
- 市场把所有人变成通才 → 通才供过于求 → 价格下跌

### 第三根支柱（尚未完全崩塌）：代码质量与架构（"taste"）

- AI agent 在保持代码整洁方面做得很差（循环依赖、重复、SOLID 失守）
- 但行业已经不在意了——**C 级/D 级代码就够了**，因为代码是写给 LLM 读的，不是给人读的
- 这项能力被压缩为一个词 **"taste"**，且价值正在缩水

### 怎么办？

- 短期保住了工作，长期不知何去何从
- 公司招聘从 "Software Engineer - Area" 变成了 "Software Engineer"——领域知识不再是差异化因素
- 考虑回大学学数学/统计/ML 去做前沿研究，但因地理和家庭因素不现实
- 半开玩笑：把木工爱好变成职业

## 二、术语补充：什么是 ADR

原文提到作者花时间"writing ADRs"，ADR = **Architecture Decision Record**（架构决策记录）。

一种轻量级文档实践，记录重要的架构决策：

1. **Context（背景）**——当时面临什么问题、什么约束
2. **Options（选项）**——考虑过哪些方案
3. **Decision（决策）**——最终选了哪个方案，以及**为什么**
4. **Consequences（后果）**——这个决策带来的影响和 trade-offs

作用是让后来的人（包括几个月后的自己）知道代码为什么长成现在这样，避免反复争论。

## 三、讨论：对文章内容与思想的看法

### 3.1 文章的真实性很高

作者给出了**具体的、可验证的证据链条**：

- 10 年金融支付领域的深耕经验
- 亲眼看着自己写 Design Docs 被经理说"太慢，用 AI"
- 亲眼看着复杂调试从 1-2 天变成一键完成
- 亲眼看着招聘描述从 "Software Engineer - Payments" 变成 "Software Engineer"

这不是在猜测，而是在描述**正在发生的事情**。

### 3.2 核心洞见：领域知识的"可提示化"

**"domain expertise 被 promptable 化"** 是一个真正的结构性变化。

过去，懂支付的工程师比不懂的贵得多，因为那层知识需要**时间和机会的积累**。现在 LLM 已经把这种知识编码进去了。一个优秀的通才 + Claude 可以在不深挖领域的情况下产出"足够好"的设计。

差距从 **"有 vs 没有"变成了"80 分 vs 95 分"**。市场对 80 vs 95 的定价差异远小于对有 vs 没有的定价差异——**这是真正的价值压缩。**

### 3.3 有所保留的地方

#### 关于"90% 的 bug 被一键解决"

剩下那 10% 的 bug 可能是真正的分水岭。那些 bug 往往不是技术上的疑难杂症，而是**和组织、业务历史、人的决策耦合在一起**的（比如一个 five-year-old workaround 源于当年某个业务经理的口头承诺）。LLM 没有对"为什么这里长这样"的隐性理解。

#### 关于"代码质量不再重要"

作者说代码写得再好也是给 LLM 读的，所以 C 级就够了。但 LLM **读烂代码的能力也没有那么强**。代码质量不会变得不重要，而是评判标准在变化——从"适合人脑认知"转向"上下文窗口友好、易于 agent 推理"。作者积累的知识需要**更新而不是丢弃**。

#### 关于"通才化 = 贬值"

作者完全忽略了自己作为高级工程师的其他价值维度：

- **我知道该 build 什么**（产品 sense）
- **我知道该说"不"**（什么不值得做）
- **我能在多个方案之间做高风险的取舍**（当失败成本很高时）
- **我能让组织买账**（沟通、信任、推动力）
- **我能对合规和审计承担个人责任**（这在金融领域很关键）

## 四、延伸讨论：新的定价框架

### 旧定价框架

```
你的价值 = Σ（你会的东西 × 市场上会这东西的人有多少）
```

- 知识稀缺 → 值钱
- 获取成本高 → 值钱
- 产出可复用 → 值钱

**逻辑：我知道你不知道的东西，而且你需要它。**

### 新定价框架

LLM 改变了知识稀缺性和获取成本，但没有改变**决策后果的真实性**。所以新框架的核心是从 **"knowing"（知道）转向 "owning"（承担）**：

```
你的价值 = 你能承担多大风险 × 你承担过多少真实后果
```

#### 维度一：从"我懂"到"我保"

LLM 没有 skin in the game——它不会因为一个 bug 被叫到合规办公室，不会被问责，不会半夜被 on-call 叫醒。**对后果负责的是人。** 这个"负责"的意愿和能力，是不可 commodity 化的。

> **不是你知道什么值钱，而是你愿意为什么负责值钱。**

#### 维度二：从"我知道怎么做"到"我知道要不要做"

LLM 擅长"给定目标，规划路径"（how）。但真正的判断在于：

- 这个功能到底要不要做？
- 做的话，80 分还是 98 分？
- 技术和合规冲突时，怎么决策？

这些答案取决于组织的风险偏好、团队能力、业务阶段、老板的信任余额等无数隐性因素。LLM 无法提供这些判断。

> **从 solution provider 升维到 decision shaper——不是给出方案，而是让决策发生且后果可控。**

#### 维度三：从"我一个人厉害"到"我能让别人厉害"

用 LLM 放大自己的判断力到更多人身上：

- 让 junior 完成过去需要 senior 才能做的任务
- 设计 prompt + 流程 + review checklist，使团队产出质量可控
- 判断 LLM 输出中"听起来对但实际上错了"的地方

> **不再用"我多厉害"衡量自己，而是用"我能让多少人达到我 80% 的水平"来定义价值。**

#### 维度四：从纯技术判断到三维判断

| 维度 | 说明 |
|---|---|
| 业务判断 | 这个技术方案在这个业务场景下未来 2 年的演进路径是什么？ |
| 组织判断 | 这个方案需要谁来配合？怎么说服他们？做砸了谁被问责？ |
| 时间判断 | 现在做还是三个月后做？砍掉哪 20% 可以节省 80% 的时间？ |

LLM 在这些维度上表现非常差，因为它的训练数据里没有"你老板昨天在会上说的话"。

> **重新定义你的"领域"——不仅是"支付系统"，而是"用支付系统支撑这个业务的组织"。**

### 新框架下的工程师画像

拿原文作者来说，按新框架他不再是"一个懂支付的工程师"，而是：

- "这个方案在我们现在的合规审查下过不了，因为第三部分的 escrow 设计引用的是 2023 年的指南，2024 年更新过了。"（**我保**）
- "如果方案 A 三周上线但需后续重构，方案 B 八周一次性搞定。以 Q4 目标，我建议方案 A，我出一份重构计划。"（**要不要做**）
- "我用 Claude 搭好框架和测试边界，你们（两个 junior）在这个边界里写逻辑，review 我来做，目标四周上线。"（**让别人厉害**）
- "我知道方案 C 技术上更优雅，但隔壁团队刚经历人员变动，核心接口三个月后才稳定，现在依赖他们有交付风险，建议方案 B。"（**三维判断**）

### 新旧框架对比总结

| 旧框架 | 新框架 |
|---|---|
| 我知道什么 | 我保证什么 |
| 我怎么做 | 我们要不要做 / 做到什么程度 |
| 我多厉害 | 我能让多少人变厉害 |
| 纯技术判断 | 技术 × 业务 × 组织三维判断 |
| 可 prompt 化 | 不可 prompt 化（因为没有 skin in the game） |

这要求作者不能只当一个技术专家，而需要变成一个**技术领导者**——即使没有管理头衔。这是他 10 年经验真正的溢价所在，而不是"知道怎么设计幂等性"。

# 延伸讨论：校招新人如何看待这个时代，如何成长

> 承接上一轮关于 "LLMs are eroding my software engineering career" 的讨论
> 关键词：AI-native 新人、架构经验积累、"taste" 的形成

---

## 六、新人面临的独特悖论

对于一个 2026 年入职的校招新人来说，他面对的局面和那位 10 年经验的作者完全不同：

| 维度 | 10 年经验作者 | 校招新人 |
|---|---|---|
| 参考系 | 经历过"没有 LLM 的世界" | LLM 从一开始就在 |
| 技能积累 | 在实战中从 0 搭过支付系统 | 可能从未独立 debug 过复杂问题 |
| 稀缺性 | 正在被压缩 | 尚未建立 |
| 焦虑来源 | 已有的积累在贬值 | 不知道该积累什么 |

新人有他独特的优势：**没有历史包袱，天然就是 AI-native。** 但也面临一个深层的悖论：

> **LLM 可以帮你做 80% 的事，但剩下那 20% 才是真正的成长发生的地方。如果你总是让 LLM 做那 80%，你永远不会拥有那 20%。**

---

## 七、"AI-native" 的优势与陷阱

### 优势

1. **生产效率极高**——一个用 Cursor/Claude Code 的新人可能比五年前一个 3 年经验的工程师产出还快
2. **试错成本极低**——可以快速搭建原型验证想法，过去需要一周的工作现在一天搞定
3. **学习路径变宽**——遇到不懂的概念，直接问 LLM 比翻文档/搜 Stack Overflow 快得多
4. **跨领域探索更容易**——前端/后端/数据/运维的边界模糊了，新人可以更早地接触全貌

### 陷阱

1. **不知道"不知道自己不知道什么"**——你让 LLM 写了一个分布式锁，它写对了，但你不理解为什么需要分布式锁。下次场景变了，LLM 可能给了一个表面上类似但实际错误的方案，你发现不了。因为你没见过"没有分布式锁会怎样"。
2. **调试能力的缺失**——调试的本质是**提出假设并验证**。如果你每次拿到 stack trace 直接扔给 LLM，你永远不会建立"根据现象猜根因"的直觉。而这种直觉在 LLM 搞不定那 10% 的时候就变成了瓶颈。
3. **架构品味的缺失**——品味来自于**看过足够多的烂代码，也看过足够多的好代码，然后形成了模式识别的直觉**。如果你只见过 LLM 生成的"中等偏上"的代码，你分不清什么是好，什么是平庸。
4. **依赖成瘾**——LLM 太好用了，以至于遇到任何问题第一反应是"问 AI"而不是"自己先想想"。这种肌肉萎缩是渐进的、无声的。

---

## 八、新的成长框架：刻意不平衡（Deliberate Imbalance）

> **日常用 LLM 最大化产出，但在关键节点刻意不用 LLM，让自己经历"从 0 到 1 的挣扎"。**

### 场景 A：全力用 AI（追求效率）

- 写单元测试、集成测试
- 写 CRUD 代码、样板代码
- 写文档、注释
- 写简单的 SQL 查询
- 写 CI/CD 脚本
- 查语法、API 用法

这些事用 AI 做你几乎不会错过任何重要的学习机会。放心用，越快越好。

### 场景 B：先用 AI，再复盘（追求理解）

- 系统设计/架构决策
- 复杂的业务逻辑实现
- 性能优化方案
- 重构方案

做法：**先让 LLM 给一版方案，逐行理解它为什么这样做，然后手动重写一遍，或者改动一个关键条件看 LLM 的方案是否需要调整。**

这个"手动重写"或"修改条件"的过程，就是架构经验积累的地方。

### 场景 C：刻意不用 AI（追求深度）

- 调试一个棘手的生产 bug
- 设计一个有争议的架构决策
- code review 中判断 LLM 生成代码的质量
- 思考"这个功能到底要不要做"

这些场景恰恰是那 10% LLM 搞不定的事。**每次你靠自己解决一个这类问题，你就在积累那 20% 的真本事。**

> 关键原则：**不要用 AI 代替你的思考，用 AI 放大你的思考。**

---

## 九、架构经验如何积累？

这是最核心的问题。在 LLM 可以随时随地给出一个"看起来合理"的系统设计的时代，新人怎么积累真正的架构能力？

### 9.1 从"评审"中学习，而非从"创作"中学习

过去学架构的路径是：
1. 看别人写的架构 → 2. 自己从 0 设计（犯错）→ 3. 被 review 打回来 → 4. 改进

现在的路径可以是：
1. **给 LLM 一个问题，让它出设计**
2. **你作为 reviewer 去批判它**——"这里为什么选 Kafka 而不是 RabbitMQ？""这个分库方案在数据倾斜时怎么办？"
3. LLM 给出理由，你判断是否合理
4. 你不确定的地方，去查资料、去问 senior、去读真实案例

这个过程中，**你虽然没有"从零设计"的痛苦，但你在练习最难的能力：判断力。** 而判断力是架构能力的核心。

### 9.2 大量读真实世界的失败案例

架构经验的本质是"知道什么会挂"。这在 LLM 时代变得更稀缺了，因为 LLM 倾向于给出**"理想情况下应该正确"**的方案，而不会主动告诉你"这个方案在什么情况下会挂"。

建议新人做的事情：

- 读 Netflix / Uber / Stripe 等技术博客中的 postmortem
- 读系统设计面试中的 trade-off 讨论，不是背答案，而是理解**为什么**
- 在公司的 on-call 中主动承担，去踩那些真实的坑
- 每次线上事故，写一份个人版的"lessons learned"

### 9.3 刻意寻找"LLM 搞不定"的场景

LLM 在以下场景中的架构能力明显偏弱，而这些正是新人可以弯道超车的地方：

| LLM 偏弱的方向 | 对应能力 | 如何培养 |
|---|---|---|
| **组织耦合** | 理解组织结构和沟通成本对技术的影响 | 多参与跨团队项目，观察决策过程 |
| **时间维度** | 知道什么该现在做，什么该以后做 | 参与长期项目，经历从设计到维护的完整周期 |
| **合规与监管** | 金融/医疗/隐私等领域的非功能性约束 | 主动接触合规类需求，别躲 |
| **遗留系统** | 在约束下的渐进式改造 | 接手一个遗留模块，别只想重写 |
| **人的因素** | 不同团队的能力差异对技术选型的影响 | 多和不同角色的人聊天，理解他们的痛点 |
| **不确定性下的决策** | 信息不足时做出可逆的决策 | 主动承担模糊的任务，练习"先做再说" |

### 9.4 建立"好代码"的参考系

因为你看不到烂代码了（LLM 不会给你写烂代码……吗？其实也会，只是烂得很平均），你需要刻意建立参考系：

- **读开源项目**——挑一个你日常使用的知名项目（比如 Redis、PostgreSQL、Kafka），看它的核心模块是怎么组织的。不一定要读通全部，重点是感受"顶级代码"是什么样的。
- **读自己的 LLM 生成的代码，然后手动重构它**——把它从 C 级改到 B 级、A 级。这个过程就是品味在形成。
- **主动 code review 别人的代码**——不只是 LLM 生成的，还有同事的。你不需要是 senior 才能提意见；你只需要说"这段我看不太懂，能不能解释一下"，往往就能发现可读性问题。

## 十、推荐的学习路径（按时间轴）

### 第 1 年：建立"可知与不可知"的边界感

- 用 LLM 最大化产出，熟悉公司技术栈
- 但每周至少留出 **2 个下午做"无 AI 日"**——纯手动写代码、调试
- 养成习惯：LLM 给出的代码，至少读一遍，不理解的地方问清楚再合
- 主动参加 on-call，尤其是那些"大家都觉得难"的 incident

### 第 2-3 年：建立判断力

- 开始 review LLM 生成的系统设计，记录你认为它"有道理"和"有问题"的地方
- 负责一个中等复杂度的模块从设计到上线到维护的完整周期
- 开始写 ADR——不是写给组织看，是写给自己看，记录"当时为什么这么选"
- 找一个 senior 定期给你做"架构复盘"，让你暴露过的思考盲区

### 第 3-5 年：建立组织影响力

- 从"用 LLM 写代码"进化到"设计一套用 LLM 写代码的流程和规范"
- 开始带更 junior 的同事，练习"让别人变厉害"的能力
- 参与跨团队的技术决策，练习多维判断
- 培养对业务的敏感度，能说清楚"这个技术方案对公司目标意味着什么"

### 第 5 年+：成为那个"说不清但就是对"的人

这时候你应该已经积累了足够的隐性判断力，成为一个组织里不可替代的"知道锅是铁打的"的人。你的价值不再来源于你比 LLM 多知道什么，而来源于：

- 你为一个决策**承担过后果**，且组织信任你
- 你**推动过复杂的事情在模糊中落地**
- 你**让周围的人变得更好**

这些是 LLM 永远做不到的。

## 十一、如何选择你的第一份工作

不同团队对 AI 的态度天差地别，而这对新人的成长轨迹影响极大。

### AI-heavy 团队（如创业公司、AI-first 产品团队）

| 方面 | 说明 |
|---|---|
| 优势 | 效率天花板极高，你能接触最新工具和流程，两年经验可能产出抵五年 |
| 风险 | 容易跳过深度思考，架构基础不牢；代码库可能被 AI 生成得混乱不堪 |
| 建议 | 适合**已经有意识建立"刻意不平衡"习惯**的人；不适合完全被动依赖 AI 的人 |

### 传统团队（如银行、大企业、合规严格的行业）

| 方面 | 说明 |
|---|---|
| 优势 | 有完善的 code review、设计评审、合规流程，逼你理解每一步为什么这么做 |
| 风险 | 进步速度慢，可能产生"我都一年了怎么还在写 SQL"的挫败感 |
| 建议 | 适合**一开始就想把基础打扎实**的人；但要注意别变成"没有 AI 就不会写代码"的反向依赖 |

### 推荐的"组合策略"

**第一份工作去一个有规范流程的传统团队**待 1-2 年，把"没有 AI 也能写好代码"的底子打牢。然后**跳到一个 AI-heavy 的团队**，把效率拉满，同时带着你已经建立的架构判断力去驾驭 AI。

这比反过来（先 AI-heavy 再传统）安全得多。因为**如果你一开始就在 AI-heavy 环境里习惯了 LLM 代劳，之后再想去传统团队补基础，你会觉得"太慢了，受不了"而放弃**。

## 十二、"AI 平庸化"陷阱与差异化策略

### 陷阱是什么

当所有工程师都在用同样的几个模型（Claude、GPT、Gemini），用同样的工具（Cursor、Claude Code、Codex），产出的代码风格、设计模式、甚至 bug 类型都趋向**同质化**。

这意味着：

- **面试官看到的候选人产出越来越像**——大家都是 LLM 写出来的代码，你怎么证明你比别人强？
- **可替代性上升**——你换一个人，用同样的 prompt，产出差不多
- **你的"AI 使用熟练度"不再是差异化因素**

### 怎么走出来

真正差异化的东西，恰恰是 LLM **做不到**或**做不好**的：

1. **能发现 LLM 的盲点**——不是"会用 AI"，而是"知道 AI 什么时候在胡说"。这种能力来自深度理解而非使用频率。
2. **能定义"什么值得做"**——产品 sense 和商业判断。LLM 不会问你"这个功能真的需要吗"。
3. **能把模糊的目标变成可执行的计划**——LLM 需要清晰的 prompt；而真实工作中没有人给你清晰的 prompt。能把"把支付体验做好"这种模糊目标拆成可落地的 step，就是巨大的价值。
4. **能在高压下做决策**——凌晨 2 点的 on-call，LLM 可以帮你查 log，但它不会替你决定"是回滚还是 hotfix"。
5. **建立信任关系**——你的老板信任你，不是因为你的代码写得漂亮，而是因为你**说能做到的，就做到了**。这个信任资本是 LLM 无法积累的。

> **不要和 LLM 比"写代码"——它已经赢了。要和它比"为什么写这个"、"什么时候不该写"、"写砸了怎么办"。**

## 十三、如何和你的 manager 沟通这套成长策略

很多新人不敢跟 manager 说"我想在某些场景不用 AI"，因为怕被认为效率低。但如果你能讲清楚逻辑，大部分好的 manager 会支持你。

### 可以这样说

> "我正在刻意练习在某些关键节点不用 AI，比如调试复杂 bug、做架构决策、code review。因为我认为这些能力是 LLM 暂时做不好的，也是我长期差异化的来源。日常的 CRUD 和测试我会全力用 AI 保证效率。这样长期看，我对团队的价值会比一个纯 AI 依赖者大得多。"

### 这样说的好处

- 表明你有**长期视角**，不是混日子
- 表明你**主动思考过自己的成长路径**
- 把这件事框成"为了团队更好"而不是"我在偷懒"
- 给了 manager 一个明确的信号：**你值得投资**

好的 manager 听到这些会很高兴；不好的 manager 如果只盯着短期产出不放，那你也提前知道这个团队不适合你长期发展。

### 同时也主动帮 manager 分担

如果你享受"刻意不用 AI"的奢侈，就需要在其他地方补偿效率。比如：

- 主动说："这部分测试代码我来用 AI 批量生成，保证覆盖率。"
- 主动说："我来搭一个团队的 prompt 模板库，让大家的 AI 产出更一致。"
- 主动说："我每周整理一份 AI 使用心得，分享给团队。"

这样你在"做深度工作"的同时，也在证明你对效率的承诺。

## 十四、如何找到有 pre-LLM 经验的 mentor

AI-native 新人最缺的就是**见过"没有 LLM 的世界"是什么样的人**。因为只有他们知道：

- 代码在没有 AI 辅助的情况下是怎么被组织起来的
- 一个复杂的系统在没有 agent 帮忙 debug 时是怎么被排查的
- 架构决策在没有 LLM 建议时是怎么被做出的

### 怎么找

1. **在团队里找到 8 年+ 经验的工程师**——不一定是你直属上司，可以是隔壁组的人
2. **真诚地请教具体问题**——不要泛泛地说"能不能带我"，而是问具体的："这段遗留代码我看了三天没看懂，你能给我讲讲它当初为什么这么设计吗？" 这种具体问题，资深工程师通常愿意花时间
3. **主动请求 code review**——跟对方说"我的代码能不能麻烦你 review 得严格一点，我想学东西"。大部分技术好的 senior 听到这个会很乐意
4. **在项目历史中寻找"遗迹"**——去看 git blame，找到那些写关键模块的人，约 coffee chat 聊当初的设计决策。这些"当初为什么这么做"的故事，是 LLM 永远不会告诉你的

### 如果没有 senior 可找

不是所有团队都有资深工程师。如果环境中所有人都跟你一样在用 AI 写代码，那你要从**外部**找：

- 读开源项目的 commit history 和 issue 讨论——那里有"为什么这样做"的真实对话
- 读技术博客和历史文章——比如 Joel on Software（虽然老但经典）、High Scalability、The Architecture of Open Source Applications
- 在技术社区（比如公司的技术分享、meetup）中主动认识不同背景的人

## 十五、一份典型"刻意不平衡"的一周样例

|  | 上午 | 下午 |
|---|---|---|
| **周一** | 全力用 AI 写本周的 CRUD 任务、测试用例、修复已知 bug | **无 AI 下午**：手动解决一个上周遗留的复杂 bug，不查 LLM，只靠 log + 代码阅读 |
| **周二** | 用 AI 辅助做系统设计 draft（场景 B：先让 AI 出方案） | 人工 review + 重构 AI 的设计方案，做 trade-off 分析笔记 |
| **周三** | 全力用 AI 处理日常需求、文档、脚本 | 参加 on-call / 处理生产 incident（如果是自己值班，刻意限制 AI 使用） |
| **周四** | 用 AI 辅助处理需求 | 约 senior coffee chat / 读一篇技术 postmortem / 写个人 ADR |
| **周五** | **无 AI 上午**：手动 code review 本周所有 LLM 生成的代码，标记"我不确定对不对"的地方 | 回顾一周：哪些 LLM 输出的东西我 review 时发现了问题？哪些东西我当时没发现但后来出事了？写进成长日志 |

关键不在于每周严格照搬，而在于**你会意识到"AI 使用"是一种可以有意识开关的选择，而不是默认打开的依赖**。

## 十六、最后说一句实话

上面这套"刻意不平衡"策略，本质上是一种**反直觉的自律**。

因为用 LLM 走捷径的感觉太好了——每个新人都会忍不住一直用。而"不用 LLM 自己死磕"的体验是痛苦的、低效的、让人怀疑自己的。

但问题是：**容易的路大家都在走，难走的路才通向稀缺。**

今天的校招新人有一个绝无仅有的机会——**你可以同时拥有 AI 的超高效率和人的深度思考**，只要你愿意在关键节点放下 AI。但如果你一直不放下，你就永远只是"一个会用 LLM 的人"。而这在 2026 年已经不稀缺了。

> 稀缺的不是"会用 AI 的工程师"。
> 稀缺的是"在 AI 的阴影下，依然能做出正确判断的人"。
