CDP: Chrome DevTools Protocol 详解 大家好，今天分享的主题是 Chrome DevTools Protocol，简称 CDP。 这是 Chrome 浏览器暴露出来的一套调试协议，所有 DevTools 的功能背后都是 CDP。更酷的是，我们也可以直接用代码通过 WebSocket 连接 CDP，实现浏览器自动化。 今天的分享会覆盖 CDP 是什么、协议原理、核心用法、实战技巧，以及和 Playwright/Puppeteer 的对比。

What is CDP

Chrome DevTools Protocol

📜 起源

  • 2012 年 Chrome DevTools 团队推出
  • 最初为 DevTools 自身设计（"吃自己的狗粮"）
  • 后来开放给第三方使用
  • 演变为远程调试标准协议

🎯 核心定位

  • JSON-RPC over WebSocket 通信协议
  • 200+ 个 Domain，700+ 个方法
  • 覆盖浏览器全部能力：DOM、CSS、Network、JS...
  • 所有 DevTools 功能 = CDP 命令的 GUI 封装

⚡ 能做什么

  • 读/写 DOM 和 CSSOM
  • 执行任意 JavaScript
  • 拦截、修改网络请求
  • 截图、导出 PDF
  • 模拟鼠标、键盘、触摸
  • 采集性能指标、覆盖率

🔧 谁在用

  • Chrome DevTools — 最直接的消费者
  • Puppeteer — Google 官方封装
  • Playwright — Microsoft 跨浏览器方案
  • Selenium — WebDriver 底层
  • VS Code / Electron — 调试器集成
CDP 最早是 Chrome DevTools 团队为自己做的协议。他们想要一个稳定、可扩展的方式来让 DevTools 和浏览器内核通信。后来发现这个协议太有用了，就开放了出来。 现在几乎所有浏览器自动化工具底层都离不开 CDP。Puppeteer 是 Google 官方的 Node.js 封装，Playwright 是微软的跨浏览器方案，连 VS Code 的调试器也是基于 CDP 的。 CDP 有 200 多个 Domain，覆盖浏览器所有能力。你做的任何事情——查看元素、网络面板、控制台——背后都是一条 CDP 命令。

Protocol · 协议细节

JSON-RPC 2.0 + WebSocket

📤 请求格式

{
  "id": 1,            // 自增 ID，匹配响应
  "method": "Runtime.evaluate",
  "params": {
    "expression": "document.title"
  }
}

📥 响应格式

{
  "id": 1,            // 匹配请求 ID
  "result": {
    "result": {
      "type": "string",
      "value": "页面标题"
    }
  }
}

⚠️ 错误响应

{
  "id": 1,
  "error": {
    "code": -32601,
    "message": "Method not found"
  }
}

📡 事件推送（无 id）

{
  "method": "Network.requestWillBeSent",
  "params": {
    "requestId": "12345",
    "request": {
      "url": "https://api.example.com/data",
      "method": "POST"
    }
  }
}

🔑 核心规则：请求含 id → 响应含相同 id · 事件不含 id · 使用前必须先 xxx.enable

CDP 使用标准的 JSON-RPC 2.0 协议，通过 WebSocket 传输。 每个请求必须有 id，服务端响应会带上相同 id 来匹配。如果请求的方法不存在或参数错误，会返回 error 对象。 除了请求-响应模式，CDP 还会主动推送事件。比如你启用了 Network 域，每次网络请求发起时 Chrome 都会推送 Network.requestWillBeSent 事件。事件没有 id，你需要根据 method 字段来区分。 有一个容易忽略的点：使用任何 Domain 之前，通常需要先调用 xxx.enable 方法来激活它。

这是 CDP 最常用的一些 Domain。 排在最前面的是 Page、Runtime、DOM、Network，这四个基本上是所有自动化任务的标配。Page 管导航和截图，Runtime 执行 JS，DOM 操作页面元素，Network 监控和拦截请求。 Input 域用于模拟用户操作，CSS 域可以读写样式。Console 和 Performance 用于调试和性能分析。Storage 管理 Cookie 和存储，Target 用来控制多标签页。 大家记住一个原则：80% 的场景用 Runtime.evaluate 就够了，剩下 20% 再去找专门的 Domain 方法。

Runtime · 瑞士军刀

Runtime.evaluate — 一行 JS 搞定一切

📖 读取数据

// 读标题
{"method":"Runtime.evaluate",
 "params":{"expression":"document.title"}}

// 读表单值
{"method":"Runtime.evaluate",
 "params":{"expression":"document.querySelector('#username').value"}}

// 读批量数据
{"method":"Runtime.evaluate",
 "params":{"expression":
   "JSON.stringify({title:document.title,url:location.href})"}}

// 读所有链接
{"method":"Runtime.evaluate",
 "params":{"expression":
   "Array.from(document.querySelectorAll('a')).map(a=>a.href)"}}

✍️ 写入操作

// 点击按钮
{"method":"Runtime.evaluate",
 "params":{"expression":
   "document.querySelector('.btn').click()",
   "userGesture": true}}

// 输入文本（触发事件）
{"method":"Runtime.evaluate",
 "params":{"expression":
   "const el=document.querySelector('#input');" +
   "el.value='hello';" +
   "el.dispatchEvent(new Event('input',{bubbles:true}))"}}

// 设置 localStorage
{"method":"Runtime.evaluate",
 "params":{"expression":
   "localStorage.setItem('token','xxx')"}}

// 异步 fetch（需 awaitPromise）
{"method":"Runtime.evaluate",
 "params":{"expression":
   "fetch('/api/data').then(r=>r.json())",
   "awaitPromise": true}}
⚡ 80% 的场景用 Runtime.evaluate 就够了，比查 CDP 文档快得多
这是 CDP 最强大的方法，没有之一。 Runtime.evaluate 可以执行任意 JavaScript 代码，返回值通过 JSON 序列化传回来。这意味着你不需要记各种 CDP 方法的参数，直接用 JS 操作 DOM 就行。 有几个关键参数需要注意：userGesture 模拟用户手势，可以触发 click 事件；awaitPromise 用于等待异步操作完成；returnByValue 控制返回值的格式。 建议：优先用 Runtime.evaluate 解决问题，搞不定再去找专门的 Domain 方法。

DOM · 文档对象模型

两种方式操作 DOM

🔵 方式一：Runtime.evaluate（推荐）

通过执行 JS 操作 DOM，简洁直观

// 获取元素文本
const title = await cdp.send({
  method: "Runtime.evaluate",
  params: {
    expression: `document
      .querySelector('.title')
      .textContent`
  }
});

// 修改样式
await cdp.send({
  method: "Runtime.evaluate",
  params: {
    expression: `document
      .querySelector('.box')
      .style
      .background = 'red'`
  }
});

🟢 方式二：DOM Domain（精确控制）

通过 nodeId 引用，适合深层遍历

// 1. 获取文档树
const doc = await cdp.send({
  method: "DOM.getDocument"
});
// → doc.result.root.nodeId

// 2. 查询元素
const el = await cdp.send({
  method: "DOM.querySelector",
  params: { nodeId: 1,
    selector: ".content" }
});
// → el.result.nodeId: 42

// 3. 获取属性
const attrs = await cdp.send({
  method: "DOM.getAttributes",
  params: { nodeId: 42 }
});
💡 建议：日常用 Runtime.evaluate 更高效；需要深层 DOM 遍历或监听 DOM 变化时用 DOM Domain
操作 DOM 有两种方式。 第一种是用 Runtime.evaluate 执行 JS 代码，简洁直观，适合大部分场景。第二种是用专门的 DOM Domain，通过 nodeId 引用元素，适合需要精确控制或深层遍历的场景。 我的建议是：80% 的场景用 Runtime.evaluate，遇到以下情况再考虑 DOM Domain：需要遍历整个 DOM 树、监听 DOM 变化（childNodeInserted 等事件）、或者需要获取计算样式。

Network · 网络层控制

网络请求监控与拦截

📡 监控请求

// 1. 启用 Network
await cdp.send({
  method: "Network.enable"
});

// 2. 收到事件（被动推送）
// ← Network.requestWillBeSent
//    {requestId, request: {url, method}}
//
// ← Network.responseReceived
//    {requestId, response: {status, headers}}
//
// ← Network.loadingFinished
//    {requestId, encodedDataLength}

// 3. 获取响应体
const body = await cdp.send({
  method: "Network.getResponseBody",
  params: { requestId: "..." }
});

🚦 拦截与修改

// 1. 设置拦截模式
await cdp.send({
  method: "Network.setRequestInterception",
  params: {
    patterns: [{
      urlPattern: "*api*",
      interceptionStage: "HeadersReceived"
    }]
  }
});

// 2. 收到拦截事件后，选择操作：
// 放行
{"method":"Network.
  continueInterceptedRequest",
 "params":{"interceptionId":"..."}}

// 修改返回值
{"method":"Network.
  continueInterceptedRequest",
 "params":{
   "interceptionId":"...",
   "rawResponse": btoa("HTTP/1.1 200 OK\r\n...")
 }}
✅ 修改请求头 ✅ 拦截返回值 ❌ 阻断请求 🔧 模拟网络条件
网络拦截是 CDP 非常强大的功能。 启用 Network 域后，Chrome 会把每个网络请求的详细信息推送过来——URL、方法、请求头、响应状态等。你甚至可以通过 getResponseBody 获取完整的响应体。 更强大的是 Network.setRequestInterception，它可以拦截请求并让你决定怎么处理。你可以放行、修改请求头、修改响应内容、或者直接阻断。这在测试 mock 数据、调试 API 接口时特别有用。 还可以用 Network.emulateNetworkConditions 模拟 3G 慢网速甚至完全断网。

CDP 提供了三种页面捕捉方式。 普通截图 captureScreenshot 截取当前视口，可以指定 clip 区域。captureFullPageScreenshot 会滚动整个页面截一张完整的长图。printToPDF 导出符合打印规范的 PDF。 还有一个录屏功能 startScreencast，它会逐帧推送 JPEG 截图，适合做实时画面录制或远程浏览器画面展示。

});
💡 实战场景：自动化登录时先用 Network.setCookie 注入 session，避免每次都走登录流程
Cookie 和存储操作是自动化测试的常见需求。 用 Network.setCookie 可以在不登录的情况下注入 session，省去登录流程。Storage.clearDataForOrigin 可以精准清除某个站点的所有数据，而不影响其他站点。 实践中，Cookie 操作建议走专门的 Network Domain，因为它提供了更多控制参数，比如 httpOnly、secure、sameSite 等。LocalStorage 和 SessionStorage 用 Runtime.evaluate 执行 JS 操作就足够了。

Ecosystem · 生态对比

CDP vs Puppeteer vs Playwright

⚡ CDP 裸协议
  • 层级：裸 JSON-RPC 协议
  • 传输：自己管理 WebSocket
  • 跨浏览器：❌ 仅 Chrome
  • 自动等待：❌ 手写
  • API 风格：JSON 命令
  • 网络拦截：⚡ 全能力
  • 学习成本：🔴 高
  • 灵活度：🔥 最高
  • 适用：深度定制、浏览器插件、底层调试
🎭 Puppeteer
  • 层级：面向对象封装
  • 传输：内置 WS 连接池
  • 跨浏览器：❌ 仅 Chromium
  • 自动等待：⚠️ 部分支持
  • API 风格：Page/ElementHandle
  • 网络拦截：✅ 基本能力
  • 学习成本：🟡 中
  • 灵活度：🟡 中
  • 适用：网页截图、导出 PDF、SSR 渲染
🎪 Playwright
  • 层级：最高层抽象
  • 传输：内置 WS 连接池
  • 跨浏览器：✅ Chrome/Firefox/WebKit
  • 自动等待：✅ 默认 auto-wait
  • API 风格：Page/Frame/Locator
  • 网络拦截：✅ route() 全能力
  • 学习成本：🟢 低
  • 灵活度：🟡 中
  • 适用：跨浏览器测试、E2E、CI/CD 集成
🔑 选型原则：做底层工具选 CDP · 简单自动化选 Puppeteer · 认真做测试选 Playwright
这三个其实不是竞争对手，而是不同抽象层次的工具。 CDP 是协议层，最灵活但也最繁琐。你需要自己管理 WebSocket 连接、处理错误重连、手动等待元素出现。适合做底层工具或需要深度定制 Chromium 的场景。 Puppeteer 在 CDP 上做了面向对象封装，隐藏了 WebSocket 细节。但只支持 Chromium，auto-wait 能力有限。 Playwright 是最高层抽象，不仅支持 Chrome，还支持 Firefox 和 WebKit。它的 auto-wait 机制是最完善的，还内置了 Trace Viewer、代码生成等测试配套工具。 选型很简单：做底层工具 → CDP，简单自动化 → Puppeteer，认真做测试 → Playwright。

Best Practices

CDP 使用最佳实践

  • ✅
    优先用 Runtime.evaluate — 80% 的场景不需要查 CDP 文档，一行 JS 搞定
  • ✅
    先 enable 再使用 — 几乎所有 Domain 使用前都需要调用 xxx.enable() 激活
  • ✅
    id 必须自增且不重复 — 每个 WebSocket 连接内 id 要单调递增，否则响应会乱
  • ✅
    awaitPromise 处理异步 — 执行 async 函数或 fetch 时，一定要设 awaitPromise: true
  • ✅
    userGesture 模拟用户 — 需要触发 click、全屏等操作时设 userGesture: true
  • ✅
    处理重连 — 页面刷新/导航后 WebSocket 会断开，需要重新连接并重新 enable
  • ✅
    HTTP API 用于管理标签页 — 新建/关闭/列出标签页走 HTTP，操作页面走 WebSocket
  • ✅
    returnByValue 控制返回值 — 默认返回引用（objectId），设 true 返回深拷贝值
  • ⚠️
    注意安全 — CDP 端口不应暴露到公网，未做鉴权的 CDP 端口可被任意控制
  • ⚠️
    调试端口只有一个 — 设置 --remote-debugging-port 后所有页面共享同一端口
最后是一些实践中总结的经验。 最核心的原则是优先用 Runtime.evaluate，它能解决绝大部分问题。其次是记住先 enable 再使用，这是最容易犯的错误。id 自增也很关键——如果 id 重复，响应就对应不上了。 awaitPromise 和 userGesture 是两个容易被忽略的参数。前者处理异步 JS，后者模拟用户手势。 安全方面要特别注意，CDP 端口完全没有鉴权机制，暴露到公网等于把浏览器控制权交给任何人。生产环境一定要做好防火墙或代理鉴权。
