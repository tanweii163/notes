// part 01 · background

Transformer LLM 的上下文窗口困境

核心挑战

  • 计算复杂度 O(n²)
    Self-attention 的计算量随序列长度平方增长，处理 100K tokens 需要巨大算力。
  • 显存容量瓶颈
    KV Cache 与激活值随长度线性增长，长序列推理的显存消耗成为实际部署的主要障碍。
  • 短窗口导致信息丢失
    当有效上下文不足时，模型无法访问完整历史，关键信息被截断丢弃。

现有方案的局限

稀疏注意力 — Sparse Attention

降低计算量，但可能漏掉关键 tokens，准确率不稳定。

位置编码外推 — Position Extrapolation

扩展位置编码范围，训练-推理分布不匹配，效果有限。

近似注意力 — Linear Attention / SSM

将 O(n²) 降为 O(n)，但性能有损失，Mamba 对长序列也出现退化。

"当我们把上下文窗口扩展到很长时，计算资源和存储容量成为硬约束。" — RCC 论文 Abstract

先说背景。Transformer LLM 在处理长文本时面临三个核心挑战。

第一，计算复杂度是 O(n²)——self-attention 的计算量随序列长度的平方增长，处理 10 万 token 需要巨大的算力成本。

第二，显存容量瓶颈。KV Cache 和激活值随长度线性增长，在实际部署中显存往往是最稀缺的资源。

第三，短窗口导致信息丢失。当有效上下文不足时，模型看不到完整的历史，关键信息被截断。

现有方案主要有三类。稀疏注意力降低计算量但可能漏掉关键信息。位置编码外推效果有限。而 Linear Attention 和 SSM 这类近似注意力的方法，把 O(n²) 降为 O(n)，但是性能有损失。

特别值得注意的是 Mamba——它本质上是一个状态空间模型 SSM，当前 token 只需要访问前一步的状态向量。但论文发现，随着上下文变长，Mamba 的性能也会退化。这说明单步状态向量能存储的信息量是有限的。

RCC 的核心洞察就来自这里。

// part 01 · key insight

核心洞察：分段压缩 + 循环拼接

🔍 从 Mamba 获得的关键启发

Mamba 中，当前 token 只需访问前一步的状态向量即可完成推理——这说明状态向量本质上是对过去信息的压缩表示。但随着序列增长，单个状态向量的容量有限，信息开始丢失。

洞察：与其让一个状态向量承担全部历史，不如分段压缩——把长序列分成固定长度的短段，每段独立压缩为紧凑向量，再循环拼接各段向量，作为完整的历史上下文。

为什么 Transformer 也能做这件事？

Mamba（SSM）视角

每个时间步 t，模型将输入压缩为状态向量 s_t = f(x_t, s_{t-1})。
推理时只需要当前 s_t，无需完整历史。
但 s_t 容量有限 → 长序列退化

Transformer 视角

可以将 Transformer 视为一种特殊的 SSM：
- 注意力机制 = 全局状态聚合
- 每层的 KV 表示 = 压缩后的历史信息
Transformer 也天然具有压缩能力

THEORETICAL MAX CONTEXT LENGTH
Max Length = Compression Rate × Encoder Window Size = 32 × 2048 = 65,536 tokens

现在是最核心的洞察部分。

Mamba 每个时间步把输入压缩成一个状态向量 s_t，等于说推理只需要当前的状态向量，不需要回看完整历史。这本身是个很聪明的设计。

但问题是，随着序列变长，单个 s_t 的容量是有限的。Mamba 自己的实验也证明了这一点——超过一定长度后性能会退化。

所以论文提出的思路是：不用一个状态向量承担全部历史，而是分段压缩。长序列分成固定长度的短段，每段单独压缩成一个紧凑的向量，然后把这些段向量拼接起来，作为完整的历史上下文。

这里还有一个关键的理论发现——Transformer 也可以被理解为一种特殊的 SSM。注意力机制本质上是对全局状态的聚合，Transformer 天然也具有压缩能力。这个认识很重要，它为 RCC 的设计提供了理论支撑。

按这个公式，最大上下文长度可以达到 32 乘以 2048，等于 65,536 个 token。

现在我们来看 RCC 的整体方法架构。

整个系统分为四步。第一步，把长文本分成固定长度的短段，每段 2048 个 tokens。第二步，用论文中设计的 Encoder 对每段进行压缩——压缩率是 32 倍。第三步，把各段压缩后的向量拼接起来，形成完整的历史上下文。第四步，把这些压缩后的向量喂给 Decoder，也就是原始的 Transformer LLM，生成最终回答。

这里有三个关键数字值得记住：32 倍压缩率——这意味着每 32 个 token 被压缩为 1 个 token；65K token 的理论最大上下文长度——这是 32 乘以 2048 的结果；在 Passkey 检索实验中，处理过的序列长度达到了 100 万个 token。

Encoder 和 Decoder 的权重都从同一个 LLM 初始化——论文用的是 Pythia-1.4B。Encoder 经过微调后专门负责压缩任务，Decoder 保持不变。

// part 03 · encoder design

Encoder 设计：从预训练 LLM 初始化

核心设计原则

  • 从同一 LLM 初始化
    Encoder 和 Decoder 共享同一个基础模型（Pythia-1.4B）的权重初始化。
  • 微调专用
    Encoder 权重在训练阶段调整，使其学习最优的压缩映射。
  • 双向注意力
    Encoder 内部使用双向注意力（Bidirectional Attention），对输入序列进行全局建模后再压缩。
  • 因果解码器
    Decoder 使用因果注意力（Casual Attention），确保生成时只能看到之前的 token。

💡 关键发现

预训练 LLM 的权重包含了丰富的语言知识和上下文理解能力，直接用它初始化 Encoder 比随机初始化效果要好得多。

Encoder 层内部结构

[ Encoder Layer i ]
Input: 2048 tokens ↓ Embedding ↓ Bidirectional Attention ↓ Feed-Forward Network ↓ LayerNorm ↓ Output: 32 compressed vectors ↓ (拼接所有层的压缩向量) ↓ Final Compressed Context

训练数据

① The Pile（约 10GB）— 通用预训练语料
② PwC Dataset — 专业领域数据
③ HotpotQA — 问答任务数据

接下来我们深入看 Encoder 的设计细节。

Encoder 的第一个设计原则是从预训练 LLM 初始化。论文发现，直接用同一个基础模型的权重来初始化 Encoder，效果远好于随机初始化。因为预训练 LLM 的权重已经包含了丰富的语言知识，Encoder 可以直接利用这些知识来做压缩任务。

第二，Encoder 内部使用双向注意力——这样在压缩前可以对整个输入序列进行全局建模，而不是只能看到前文。

第三，Decoder 侧保持使用因果注意力——生成时只能看到之前的 token，这是语言模型的标准设置。

训练数据有三个来源：The Pile 是通用预训练语料，大约 10GB；PwC 是专业领域数据；HotpotQA 是多跳问答数据集。论文也在 Limitations 中坦诚说明，受限于训练数据集，目前 RCC 只支持 QA 任务。

循环压缩机制：分段 → 压缩 → 拼接

RECURRENT COMPRESSION FLOW — FOR SEQUENCE OF 6144 TOKENS

Chunk 3: 2048

ENCODER
v₁
v₂
v₃

[v₁, v₂, v₃] → 压缩后序列: 3 × 32 = 96 tokens（而非 6144）
↓

Transformer LLM Decoder → 生成回答

压缩过程详解

  • 固定窗口长度 — Encoder 每次处理固定 2048 tokens，超长序列自动循环分段。
  • 固定压缩率 — 每 2048 tokens 压缩为 64 tokens → 32x（论文中称 mem_len=32）。
  • 无信息丢弃 — 拼接所有段向量 = 完整历史，无截断损失。
  • Transformer 解码 — Decoder 可在任意位置访问压缩向量。

与标准方案的对比

方案显存复杂度最大长度信息保留
标准 Transformer O(n²) 受显存限制 完整
Mamba / SSM O(n) 退化 部分丢失
RCC（ Ours） O(n·C) 65K+ 完整

这张图展示了循环压缩机制的完整流程。以 6144 个 token 的序列为例。

第一步，自动分段。由于 Encoder 的固定窗口是 2048 tokens，6144 会分成三段：Chunk 1、Chunk 2、Chunk 3。

第二步，逐段压缩。Encoder 把每段 2048 tokens 压缩为一个 64 维的向量 v1、v2、v3。

第三步，拼接。把这三个向量拼接起来，形成完整的压缩上下文。注意这里压缩后只有 3 乘以 64 等于 192 个 token，而非原来的 6144——压缩率接近 32 倍。

第四步，Decoder 处理压缩后的向量，生成最终回答。

右边的对比表也值得注意：标准 Transformer 是 O(n²) 但显存受限；Mamba 是 O(n) 但对长序列会退化；RCC 在 O(n·C) 的复杂度下同时实现了最大长度扩展和信息完整保留。

// part 03 · instruction reconstruction

指令重建方法：解决双重压缩问题

⚠️ 问题：指令 + 上下文同时压缩时性能退化

在实际应用中，用户输入通常包含两部分：指令（Query）和上下文（Context）。当两者都被压缩时，Decoder 会面临双重退化：既要从压缩后的上下文检索信息，又要从压缩后的指令理解任务意图，导致生成质量显著下降。

问题分析

Case A（仅压缩 Context）
User: "Where does the cat sit?"
Context: [压缩后的猫坐在沙发上的内容]
→ Decoder 正常理解 Query，检索 Context ✓
Case B（同时压缩 Query + Context）
User: [压缩后的"Where does..."]
Context: [压缩后的猫坐在沙发上的内容]
→ Decoder 无法理解任务，输出质量差 ✗

解决方案：指令重建

Instruction Reconstruction Method
  • 保留原始 Query
    用户指令保持原始格式，不经过 Encoder 压缩，直接作为 Decoder 的独立输入。
  • 压缩 Context
    长文档、长对话历史等上下文信息经过 Encoder 压缩。
  • 分离注入
    压缩后的上下文向量和原始 Query 分别注入 Decoder 不同位置。
// 伪代码
compressed_ctx = encoder(long_context)
orig_query = tokenizer(user_prompt)
response = decoder(orig_query + compressed_ctx)
"我们发现，当指令和上下文都被压缩时，模型性能显著退化。指令重建方法通过保留原始 Query，有效缓解了这一问题。" — RCC 论文 Section 3.2

这一页讲的是论文中一个很实用的发现和解决方案。

问题出在什么地方？实际应用中，用户输入通常包含两部分：指令和上下文。当两者都经过 Encoder 压缩时，Decoder 面临双重困难——既要从压缩后的上下文检索信息，又要从压缩后的指令理解任务意图。

论文里用了一个具体例子：用户问"猫坐在哪里？"，上下文是描述猫坐在沙发上的长段文字。如果只压缩上下文不压缩指令，模型可以正常理解问题并检索。但如果把问题和上下文同时压缩，模型甚至无法理解用户想问什么。

解决方案很简单但很有效——保留原始 Query 不压缩，只压缩长上下文。这就是论文里说的指令重建方法。

从伪代码可以看到：压缩后的上下文和原始 Query 分别注入 Decoder，原始 Query 保持完整语义，压缩后的上下文提供长历史信息。

// part 03 · theory

理论视角：Transformer 作为特殊的 SSM

SSM（状态空间模型）的压缩特性

// Mamba: SSM 前向传播
h_t = Aht-1 + Bxt // 状态更新
y_t = Ch_t // 输出映射
其中 A, B, C 为可学习矩阵
每步计算: O(1) 状态 + O(n) 线性增长
问题: 单个 h_t 容量有限 → 长序列信息丢失

🔑 核心洞察

Transformer 的注意力机制可以理解为一种全局状态聚合——每个位置的 Query 与所有 Key 做内积，相当于对历史状态做加权求和。因此 Transformer 在理论上也是一种 SSM，只是状态表示的维度更高、表达能力更强。

Transformer 的 SSM 视角

// Transformer: Attention 前向传播
Attention(Q,K,V) = softmax(QKT/√d)·V
本质上:
每步访问 所有历史 KV
全连接状态聚合
= 高维 SSM with 全局感受野
✓ Transformer 具有天然的压缩能力

RCC 的理论优势

🧠
更强状态表示 — Transformer 比 SSM 有更大的状态维度，压缩能力更强。
🔄
循环拼接保留全历史 — SSM 单步一个状态向量，RCC 用多个压缩向量拼接，容量更大。
📚
继承预训练知识 — 从 LLM 初始化，利用预训练权重中的语言理解能力。

这一页从理论角度解释为什么 RCC 有效。

先看 SSM 的视角。Mamba 的前向传播是 h_t 等于 A 乘以 h_{t-1} 加上 B 乘以 x_t，再输出 y_t 等于 C 乘以 h_t。这个框架下每步计算是常数级别的，但问题是单个 h_t 状态向量能存储的历史信息有限。

关键洞察来了——Transformer 其实也可以被理解为一种 SSM。注意力机制本质上是对所有历史 Key-Value 信息的加权求和，每个位置的 Query 访问所有历史状态。这相当于一个拥有全局感受野的高维 SSM。

所以论文的核心论点是：Transformer 天然具有压缩能力，只是传统 Transformer 没有把这个能力显式地建模为压缩任务。RCC 实际上是在"激活"并"优化"这个能力。

三个理论优势：第一，更强的状态表示维度；第二，循环拼接避免了单状态向量的容量瓶颈；第三，从预训练 LLM 初始化可以直接利用已有的语言知识。

在长文本中随机插入一个关键词（如 "magic word: xyz"），要求模型在压缩后的上下文中准确定位并输出该关键词。这是测试精确信息检索能力的标准任务。

KEY RESULTS SUMMARY
BLEU4 ≈ 0.95（32x 压缩）  |   Passkey 精度 ≈ 100%（1M tokens）  |   显著节省显存

来看实验结果，首先是文本重建和 Passkey 检索两个任务。

文本重建是在 32 倍压缩率下测试模型能否从压缩后的表示重建原始文本。BLEU4 是衡量 n-gram 重叠度的指标，满分是 1。RCC 达到了 0.95——这意味着即使压缩了 32 倍，重建出来的文本和原始文本在词汇和结构上高度相似。基线方法如 StreamingLLM 约 0.78，Landmark 约 0.85，都明显低于 RCC。

Passkey 检索任务是在一个 100 万 token 的超长序列中插入一个隐藏的关键词，然后问模型这个关键词是什么。这是测试精确信息检索能力的标准任务。RCC 在 1M tokens 序列上达到了接近 100% 的检索精度。

这两个实验放在一起说明：32 倍压缩后，压缩后的向量既保留了整体语义（可以重建文本），又保留了精确信息（可以检索关键词）。这是非常强的结果。

再看第二组实验结果——长文本问答和显存分析。

长文本问答上，RCC 和原始未压缩的 Pythia 相比，在各个长度段都保持了持平的性能。特别值得注意的是最后一行的超长文本 QA——原始 Pythia 在这个长度已经 OOM 了，但 RCC 仍然可以运行。这是 RCC 的核心优势。

显存对比表更直观。随着序列增长，原始 Pythia 的显存消耗急剧增加，32K tokens 时就 OOM 了。RCC 的显存消耗增长明显更慢，在 16K 时比原始方案节省约 70%，在 32K+ 时原始方案已经完全无法运行，而 RCC 还可以继续工作。

总结这两组实验：RCC 在性能上与未压缩方案持平，在显存效率上显著优于原始方案，使得在有限显存下处理超长序列成为可能。

⚡

这张图详细展示了 RCC 的架构实现细节。

左上是 RCC 层内部的 Encoder Block。前向传播的流程是：输入经过双向自注意力计算，Q、K、V 都来自同一个输入 X，然后用标准的 softmax 注意力得到 H，接着做残差连接和 LayerNorm，再经过前馈网络 FFN，最后通过一个压缩投影层，把 hidden 维度映射到压缩后的 mem_len 维度。

右下是显存节省的数学原理。原始 Transformer 的 KV Cache 是 O(n·d·layers)，激活值是 O(n²·layers)。RCC 把 n 变成了 n/32——因为输入序列被压缩到了原始长度的三十二分之一。

实际测试中，16K tokens 长度时显存节省约 50-70%。这在边缘设备或显存受限场景下非常有价值。

最后，RCC 使用了 FlashAttention-2 来加速注意力计算。FlashAttention 是一个 IO-aware 的优化实现，通过矩阵分块计算显著减少了显存访问次数，在长序列场景下可以提速 2-4 倍。

// part 05 · limitations

局限性与未来方向

当前局限性

① 任务范围受限

由于训练数据的限制，RCC 目前仅支持 QA 任务。无法直接应用于对话生成、代码生成、创意写作等场景。需要更多样化的训练数据来提升泛化能力。

② 模型规模较小

目前只发布了基于 Pythia-1.4B 的模型（14 亿参数），在参数规模和能力上远弱于当前主流的 Llama、Qwen、Mistral 等模型。需要在大模型上验证效果。

③ 超长序列性能退化

当推理序列长度超过训练数据长度时，模型性能可能会下降。RCC 的训练数据覆盖范围制约了其实际应用中的有效长度上限。

未来工作计划

🗂️ 更多基础模型

计划发布基于 Qwen2、Llama3、Mistral 等更强开源模型的 RCC 版本，覆盖不同参数规模和架构。

🎯 更广泛任务支持

引入更多样化的训练数据，支持 对话生成、代码补全、摘要生成、翻译 等多种任务，扩展 RCC 的实际应用场景。

📐 自适应压缩率

当前压缩率固定为 32x。未来可探索 内容自适应的压缩率——对简单重复内容使用更高压缩率，对信息密集内容使用较低压缩率。

⚙️ 更高效的架构

探索更轻量的压缩编码器，减少 Encoder 层的参数量，降低训练和部署成本，使 RCC 更容易集成到实际系统中。

OPEN QUESTIONS
如何将 RCC 泛化到任意文本任务？  |   如何在更大模型上保持压缩效率？  |   能否与 KV Cache 量化方法（如 PolyKV）结合？

这一页客观地总结论文的局限性和未来方向。

当前主要局限有三个。第一，任务范围受限。由于训练数据只包含 QA 类数据，目前 RCC 无法直接用于对话生成、代码生成等场景。第二，模型规模较小。只发布了基于 Pythia-1.4B 的版本，远小于主流模型的规模，效果的可迁移性有待验证。第三，超长序列性能退化。当推理长度超过训练数据覆盖范围时，性能会下降。

未来工作计划列了四个方向。最值得关注的是第一点——论文作者明确表示会发布基于 Qwen2、Llama3、Mistral 等更强模型的版本。另外，动态压缩率也是一个有趣的方向——让模型自己判断什么时候该高压缩、什么时候该低压缩。

最后有一个值得思考的问题：RCC 的语义压缩方法能否和 PolyKV 这类 KV Cache 量化方法结合？两者从不同维度压缩——RCC 是语义层压缩，PolyKV 是数值层量化——理论上可以叠加，效果值得探索。

论文核心贡献

01

循环压缩框架

提出 Encoder-Decoder 循环压缩框架，通过分段压缩 + 状态向量拼接，在有限显存下高效扩展上下文窗口至 65K+ tokens。

02

32x 压缩性能

32 倍压缩率下 BLEU4 ≈ 0.95（文本重建），1M token Passkey 检索精度 ≈ 100%，同时保持长文本问答性能与未压缩模型持平。

03

指令重建方法

提出指令重建方法，通过保留原始 Query、仅压缩 Context，有效解决了双重压缩导致的性能退化问题。

对 Agent 对话压缩的启发

  • 语义压缩 vs 数值压缩
    RCC 是语义层压缩（保留语义），可与 PolyKV 等 KV Cache 量化（数值层）叠加使用，理论上压缩效果可以叠加。
  • 分段循环设计
    Agent 的对话历史天然是分段的（每个 turn 是一段），可以直接借鉴"分段压缩 + 向量拼接"的设计思路。
  • 指令保留原则
    用户最新指令不应被压缩（必须保持完整语义），压缩的对象应该是历史对话内容。
  • 端到端训练
    Encoder-Decoder 架构支持端到端优化，可以针对 Agent 的具体任务场景做定制化训练。

对 Agent 对话压缩的启发，我认为有四点最值得参考。
