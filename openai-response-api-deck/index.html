03 · 概念

什么是 Response API?

OpenAI 于 2025 年推出的新一代 API，统一了文本生成、工具调用、内置搜索和代码执行等能力。一次请求，一个 Response 对象返回所有结果。

单一端点，多个能力

POST /v1/responses — 替代 chat/completions、assistants、batch 的多个端点

内置工具即服务

联网搜索、文件检索、操作电脑、代码执行——不再需要自己编排第三方服务

流式与非流式统一

同一个 API，stream: true/false 切换，返回格式一致

# 最简调用 curl https://api.openai.com/v1/responses \ -H "Authorization: Bearer $OPENAI_API_KEY" \ -H "Content-Type: application/json" \ -d '{ "model": "gpt-4o", "input": "What is the latest LLM paper?" }' // 返回一个 Response 对象 { "id": "resp_abc123", "status": "completed", "output": [{ /* message(s) */ }], "usage": { /* tokens */ } }

04 · 动机

为什么需要 Response API?

减少样板代码

Chat Completions 做一次搜索要手动调 API、解析 JSON、处理 tool_calls 循环。Response API 内置了这些流程。

统一的工具生态

以前要拼接 Slack bot + SerpAPI + 自建 RAG + 沙箱。现在一个 API 涵盖了这些内置工具，且由 OpenAI 维护。

更简单的状态管理

每次返回完整的 Response 对象，不再需要自己维护 conversation history 或 thread 状态。前后端通信更清晰。

支持多模态输入

文本、图片、音频均可直接传入 input 字段。不再需要 base64 编码或多段 content 数组。

可观测性提升

每个 Response 自带 usage、tool_use 统计，方便追踪成本和行为。

面向 Agent 设计

Response API 专为 agentic 工作流打造——支持多步工具调用、中间状态可见、易集成到循环中。

05 · 对象模型

核心对象

Response

id — 唯一标识
model — 使用的模型
status — completed / failed / in_progress
output — 输出项数组（文本 / tool_call / 图片等）
tools — 本次启用的工具列表
usage — token 消耗明细

Tool

type — web_search / file_search / computer_use / code_interpreter
config — 工具特定配置（搜索源、文件 ID 等）

ToolCall

id — tool call 唯一 ID
type — 工具类型
output — 工具执行结果

参考: platform.openai.com/docs/api-reference/responses

06 · 内置工具一览

四大内置工具

🔍 web_search

联网实时搜索，基于 Bing。可指定搜索源（internet / news / academic），返回带引用的结果片段。
需要 user_context 授权

📁 file_search

在已上传的文件（PDF、DOCX、代码等）中检索。自动 chunk + embedding，支持交叉引用。
替代 Assistants 的 file_search

🖥️ computer_use

模型可以"看"屏幕截图并执行鼠标/键盘操作。适合 UI 自动化测试、数据录入等场景。
beta · 需 display environment

📊 code_interpreter

在沙箱 Python 环境中执行代码，支持 matplotlib、pandas、numpy。结果自动嵌入 Response。
源自 Assistants Code Interpreter

可以同时启用多个工具，模型会自动决定何时调用哪个工具。

07 · 代码示例 · web_search

联网搜索

启用 web_search 后，模型会在需要最新信息时自动触发搜索，结果以引用形式嵌入到输出中。

✅ 无需手动调用搜索 API
✅ 自动生成引用标注
✅ 支持筛选搜索源
✅ 可控制搜索深度
import openai client = openai.Client() response = client.responses.create( model="gpt-4o", input="What happened at OpenAI DevDay 2025?", tools=[{ "type": "web_search", "search_context_size": "medium" }] ) # output 中自动包含搜索结果 + LLM 总结 print(response.output[0].content) # 查看引用的来源 for citation in response.citations: print(citation.url)

08 · 代码示例 · file_search

文件检索

先上传文件，然后在 Response API 中引用。模型会自动在文件内容中搜索相关片段来回答。

支持的文件类型

PDF DOCX TXT MD Python JSON CSV
# 先上传文件 file = client.files.create( file=open("research-paper.pdf", "rb"), purpose="assistants" ) # Response API 中引用 response = client.responses.create( model="gpt-4o", input="Summarize section 3 of the paper", tools=[{ "type": "file_search", "file_ids": [file.id] }] ) print(response.output[0].content)

09 · 代码示例 · computer_use

操控电脑

模型获取屏幕截图，分析界面元素，然后返回操作指令（点击、输入、滚动等）。适合 UI 自动化。

⏺ 模型接收 1280×720 截图
🖱 返回动作: click(x,y) / type(text) / scroll(dx,dy)
🔄 循环: 截图 → 分析 → 执行 → 再截图
response = client.responses.create( model="computer-use-preview", input=[{ "role": "user", "content": "Open Chrome and go to openai.com" }, { "type": "computer_screenshot", "image": base64_screenshot }], tools=[{ "type": "computer_use", "display_width": 1280, "display_height": 720 }] ) # 提取模型建议的操作 for item in response.output: if item.type == "computer_action": print(item.action) # 例如: {"type":"click","x":640,"y":480}

10 · 架构对比

Chat Completions vs Response API

维度Chat CompletionsResponse API
端点 POST /v1/chat/completions POST /v1/responses
工具调用 需手动完成 tool_calls 循环 内置自动编排，模型自己决定
联网搜索 需自建或集成第三方搜索 API 内置 web_search，一行启用
文件检索 需自行 chunk + embedding + 检索 内置 file_search，上传即用
多模态输入 content 数组，base64 编码 直接传入文本/图片/音频
状态管理 需自行维护 messages 数组 Response 对象自带完整上下文
适用场景 简单对话、纯文本生成 Agent、搜索、自动化、多工具编排
结论: Chat Completions 不会消失，但所有新功能都在 Response API 上首发。

11 · 最佳实践

用好 Response API 的几点建议

1. 善用 previous_response_id

替代手动拼 messages 数组。传入上一轮的 response id 即可维持对话上下文，更简洁且不会遗漏。

2. 精确配置 tool 参数

web_search 指定 search_context_size，file_search 精确到 vector_store。不必要的搜索浪费 token。

3. 使用 stream 提升体验

流式模式下 Server-Sent Events 逐块返回，用户能实时看到内容生成。同一个 Response 对象格式不变。

4. 监控 usage 字段

每个 Response 都包含详细 token 使用和工具调用次数。用于成本审计和行为分析。

5. 组合多个工具

一次请求可同时启用 web_search + file_search + code_interpreter。模型会按需编排调用顺序。

6. 处理错误状态

关注 status: failed 和 error 字段。内置工具可能因权限或超时失败，需要有降级策略。
