02 // three-way tradeoff

延迟 · 可靠性 · 成本：三个维度

任何 LLM Agent 工作流的部署都面临这三个指标的基本权衡（fundamental tradeoff）。

⏱️

延迟（Latency）

从请求发出到收到最终结果的总耗时

网络传输τ（随机）
推理计算X / λₜₕ
结果生成L / λᵍᵉⁿ
🔴 硬约束：SLA / 用户体验
✅

可靠性（Reliability）

输出可接受的概率

推理质量1 - e^{-αX}
表达质量1 - e^{-βL}
整体乘积形式
🟢 目标：最大化
💰

成本（Cost）

运行工作流的经济代价

输出计费c_tok · L
计算资源c_comp · (X+L)
API 调用按次计费
🔵 硬约束：预算上限

02 // latency deep-dive

延迟从哪里来？

LLM Agent 的一次执行包含两个串行阶段，每个阶段的处理速度不同。

stage 1
推理（Thinking）
内部 Chain-of-Thought / 推理过程
X 个 token · 速率 λₜₕ
→
stage 2
生成（Generation）
输出最终回答
L 个 token · 速率 λᵍᵉⁿ
总延迟
T = τ + X / λₜₕ + L / λᵍᵉⁿ
τ 是随机变量（网络+编排），推理和生成阶段是确定性的（由 token 数和速率决定）

// 关键速率对比

λₜₕ推理阶段的 token 处理速率（通常较慢）
λᵍᵉⁿ生成阶段的 token 输出速率（通常较快）
τ基础设施延迟 — 网络、排队、编排开销

💡 关键洞察
生成阶段速率 λᵍᵉⁿ 和推理阶段速率 λₜₕ 在 vLLM / TensorRT-LLM 等推理引擎中的差异可达 2-5×。论文参考了 Kwon et al. (2023) PagedAttention 的工作。

02 // reliability deep-dive

可靠性为什么是"指数形式"？

论文用参数化指数函数建模可靠性，反映了一个普遍现象：边际收益递减。

可靠性函数
ρ(X, L) = (1 - e^{-αX}) × (1 - e^{-βL})
📉
边际收益递减：给 Agent 前 200 个 token，可靠性从 0% 升到 60%；再给 200 个 token，只到 80%；再给 200 个，只到 90%……
🔬
α 和 β 的角色：α 控制推理 token 的效率，β 控制输出 token 的效率。值越大 → 越"高效"。

// 指数曲线可视化

β=0.0005
(低效)
1,000 tok → 39%
β=0.001
(中等)
1,000 tok → 63%
β=0.002
(高效)
1,000 tok → 86%
β=0.003
(很高)
1,000 tok → 95%

同样 1,000 个输出 token，β 不同的 Agent 可靠性差异巨大

🍔 包子类比
吃第 1 个包子很管用，吃第 10 个就没那么管用了。
β 就是"胃口大小"——β 小的人胃口大，需要更多包子。

02 // cost deep-dive

成本：两顶帽子

论文区分了用户可见成本和系统计算成本——两顶帽子往往是不同的。

👤 用户可见成本（User Cost）
C^u_j(L_j) = c_tok · L_j
按输出 token 计费。这是用户直接看到的账单。
例：GPT-4o $2.50/1M 输出 token，L=2000 → $0.005
⚙️ 系统计算成本（Compute Cost）
C^c_j(X_j, L_j) = c_comp · (X_j + L_j)
推理和输出 token 都需要算力。
c_comp 取决于模型参数规模、硬件、以及推理引擎效率

// 计算成本估算

论文引用了 Kaplan et al. (2020) 的 Scaling Laws，给出了 FLOPs 的计算公式：

FLOPs_per_token = 2β + 2·n_layer·n_ctx·n_attn

其中 β 是模型参数数量，n_layer/n_ctx/n_attn 是 Transformer 架构维度。

实际含义：更大的模型 → 每个 token 更贵。这也意味着"给 Agent 更多 token"的成本在大型模型上更高。

02 // synthesis

三者的关系：不可能三角

⏱️ 延迟
给更多 token → 更可靠但更慢
三者
无法
兼得
✅ 可靠性
需要更可靠 → 必须增加 token → 成本↑延迟↑
💰 成本
压缩成本 → 减少 token → 可靠性↓

论文的博弈视角

X （推理 token） 增加 → 推理更好但系统成本↑
L （输出 token） 增加 → 表达更好但用户成本↑且延迟↑
🎯
论文聚焦：在 X 固定、L 可变 的条件下，如何在延迟和成本约束下最优分配 L？

💡 这类似于经济学中的效率-公平权衡——资源有限，如何分配才最优？

03 // llm agent model

LLM Agent：两阶段执行模型

论文对 LLM Agent 的建模非常精炼——仅用两个变量 X 和 L 就抓住了核心。

🔍 推理阶段
X_j —— 推理 token 数
表示 Agent 在内部进行思考、推理、规划的计算量。对应 Chain-of-Thought 中的"思考步骤"。
📤 生成阶段
L_j —— 输出 token 数
表示最终输出给用户或下游 Agent 的文字长度。也是 API 计费的依据。
💡
论文核心假设：X 由 Agent 内部策略决定（如模型自带的 max_thinking_tokens），系统设计者主要控制 L。

// 完整的三元组

组件对延迟的影响对可靠性的影响对成本的影响
τ (基础设施)✅ 直接加❌❌
X (推理 token)✅ X/λₜₕ✅ 1-e^{-αX}✅ c_comp·X
L (输出 token)✅ L/λᵍᵉⁿ✅ 1-e^{-βL}✅ c_tok·L+c_comp·L

🔑 简化之美
整个 LLM Agent 的性能只需 X, L, α, β, λₜₕ, λᵍᵉⁿ, τ, c_tok, c_comp 这 9 个参数就能刻画。这使得后续的优化成为可能。

03 // latency model

延迟模型：三部分构成

延迟 = 基础设施 + 推理计算 + 结果生成。前两部分与输出长度无关。

τ (基础设施延迟)
网络传输、请求排队、编排开销。论文将其视为随机变量，使用均值 τ̄ 参与优化。
X / λₜₕ (推理时间)
内部推理 token 的生成时间。推理阶段的生成速率通常较慢（因为是自回归+KV cache 管理）。
L / λᵍᵉⁿ (生成时间)
输出 token 的生成时间。这是系统设计者可控制的部分——通过调整 L 来平衡延迟。

// T_fixed 的概念

论文定义了一个重要概念——固定延迟 T_fixed：

T_fixed = Στ̄ + Σ(X/λₜₕ)

这是与输出长度 L 无关的部分。延迟约束 T 减去 T_fixed 就是可用于输出生成的"时间预算"B_T。

💡 关于 τ 的随机性
论文将 τ 视为随机变量，但在优化中使用其均值 τ̄。这隐含假设了优化对不确定性是"风险中性"的。对于风险厌恶的场景（如 P99 延迟），可能需要不同的建模方法。

03 // reliability model

可靠性模型：为什么乘积形式？

可靠性 = 推理正确 AND 表达正确 → 自然就是乘积。

可靠性函数
ρ_j = (1 - e^{-α_j X_j}) · (1 - e^{-β_j L_j})
✅
推理部分 (1-e^{-αX})：推理做得越好、考虑得越全面，越可能得出正确的内部结论
✅
表达部分 (1-e^{-βL})：输出越长越可能完整、清晰、无歧义地传达结果
📐
乘积形式：两者缺一不可——推理对了但表达错了，或者表达对了但推理错了，整个输出都不合格

// 合理性论证

❶ 值域 [0,1]：两个因子都在 0~1 之间，乘积自然也是概率
❷ 单调性：X 或 L 增加时，可靠性严格递增
❸ 凹性：边际收益递减，符合直觉
❹ 自由度：α 和 β 为每个 Agent 提供个性化校准
❺ 数学友好：log(1-e^{-βL}) 是凹函数，可用凸优化

📊 对比其他选择
为什么不选 sigmoid？因为 sigmoid 在 L=0 时不为 0（不合理），而指数形式 ρ(0)=0。
为什么不选线性？线性无法体现边际收益递减。

03 // intuition

指数形式的直觉：边际收益递减

选指数函数不是偶然的——它完美刻画了"投入-产出"的饱和效应。

第1个 200 token → 可靠性 39%
第2个 200 token → 可靠性 23% ↑ (累计62%)
第3个 200 token → 可靠性 14% ↑ (累计76%)
第4个 200 token → 可靠性 7% ↑ (累计83%)
📊
现实类比：背单词、健身、编程——任何技能学习都遵循"学习曲线"（power law / exponential saturation）

// 参数 β 的实际意义

β 值到 80% 可靠需多少 token？类型
0.00053,219🍱 大胃口
0.00101,609🍜 中等
0.0020805🍚 小胃口
0.0030536🥄 极小胃口

到 80% 可靠需要的 token 数 ≈ -ln(0.2) / β
β 越大 → 需要的 token 越少 → Agent 越"高效"

🎯 这解释了后面的注水分配：
β 小的 Agent 需要更多 token，β 大的只需要一点

03 // non-llm agents

非 LLM Agent：确定性模块

论文对非 LLM Agent 的建模简单直接——固定参数，不可优化。

非 LLM Agent 的表示
(T_i, ρ_i)
一对固定参数：延迟和可靠性
🗄️
数据库查询
延迟取决于查询复杂度+索引，可靠性接近 1（正确数据+正确 SQL）
🔢
优化求解器
延迟取决于问题规模，可靠性取决于算法（确定性的通常 ρ=1）
✅
校验器 / 测试运行器
延迟取决于运行时间，可靠性取决于测试覆盖率

// 论文的建模局限

Non-LLM 模块被处理为黑盒（T_i, ρ_i）。这在某些场景下过于简化：

⚠️ 非 LLM 模块也可能有资源-质量权衡（如精确求解 vs. 近似求解）
⚠️ 某些校验器可多次运行提升置信度——这引入了额外的可优化维度

💡 工程建议
在实际系统中，即使是非 LLM 模块，也可以为其拟合一个"可靠性提升曲线"——比如运行 1 次测试 vs. 运行 3 次的 bug 检出率。

04 // workflow structures

顺序结构：串联的数学

顺序是最简单也最基础的结构——一个 Agent 的输出成为下一个的输入。

Agent 1
→
Agent 2
→
Agent 3
→
⋯
→
Agent n
顺序延迟
T_seq = Σ T_k
必须等上一个完成才能开始下一个——延迟累加
顺序可靠性
R_seq = Π ρ_k
所有 Agent 都成功 → 整体才成功——可靠性相乘

⚠️ 顺序的关键问题

📉 误差累积：前一个 Agent 的小错会被后一个放大
🐢 延迟大：n 个 Agent 的延迟全部累加
📊 可靠性乘法效应：每个 Agent 99% → 10 个串行后只有 90.4%；每个 90% → 10 个后只有 34.9%

🎯 这就是为什么论文聚焦顺序！顺序工作流的优化最具挑战性，
也是后续注水分配的直接应用场景

04 // parallel composition

并行结构：合取 vs. 冗余

并行有两种模式——取决于是否需要所有结果。

AND（合取）
T = max(T_i, T_j)   R = ρ_i · ρ_j
需要所有分支都正确。延迟取决于最慢的，可靠性仍然是乘积。
OR（冗余）
T = max(T_i, T_j)   R = 1 - (1-ρ_i)(1-ρ_j)
任意一个分支正确即可。延迟仍然取决于最慢的，但可靠性变成"至少一个成功"。
💡
OR 模式相当于冗余执行（redundancy）：让 2 个 Agent 做同一件事，只要一个对就行。这是提升可靠性但增加成本的常见策略。

// 冗余 vs 合取示例

🌐 合取：搜索结果+知识库
既需要搜索引擎结果正确，又需要知识库检索正确，两者都对才行
R = ρ_search · ρ_kb
🔄 冗余：两个 LLM 各自推理
问两个模型同一个问题，选一个答案——只要有一个对就行
R = 1 - (1-ρ_modelA)(1-ρ_modelB)

📊 OR 模式在 Agent 可靠性低时特别有效
例：2 个 70% 可靠的 Agent 做 OR → 整体可靠性 91%

04 // feedback loops

反馈循环：迭代的力量

反馈循环是多轮交互——Agent i 和 j 交替执行 K 次迭代（如：生成→评审→修改→评审⋯）。

Agent i
生成
→
Agent j
评审
→
Agent i
修改
→
⋯
K 轮后
反馈延迟
T_fb = K · (T_i + T_j)
K 轮迭代 × 每轮两阶段
反馈可靠性
R_fb = (ρ_i · ρ_j)^K
每轮都正确 → 整体才正确
⚠️
反馈循环的可靠性随 K 指数衰减！如果 ρ_i·ρ_j < 1，K 越大整体可靠性越低。这也是为什么很多实践限制迭代次数。

// 何时反馈有效？

✅
有效：修正错误
Agent i 初版有 bug，Agent j 指出，i 修正后正确。只要 j 评审可靠且 i 能改对，反馈提升质量。
❌
无效：越改越错
如果评审者 j 不可靠，把对的改成错的——迭代反而降低质量。

💡 论文的简化假设
论文假设每轮独立且每轮必须全部正确。实际上有效的反馈会在修正后提高后续概率——但会使模型复杂化。

04 // modular reduction

模块化规约：任何结构都可以变顺序

论文的一个关键论点：研究顺序工作流不失一般性——因为复杂结构可以归约为顺序组合。

🔀 并行子工作流 → 等效模块 (T_module = max(T₁…T_m))
↓
🔁 反馈循环 → 等效模块 (T_module = K·(T_i+T_j))
↓
📋 所有模块串联 → 一个顺序工作流

// 规约示例

原始结构
A → (B ‖ C) → D，其中 B 和 C 并行(合取)
↓ 将 (B ‖ C) 视为模块 M
规约后
A → M → D，即 3 个阶段顺序

⚠️ 需要注意
规约的前提：子结构内部的延迟/可靠性可聚合为等效参数。
且聚合后失去了内部结构信息——你无法再优化子结构内部。

05 // optimization core 🎯

优化问题形式化

论文的优化问题是一个在约束下最大化可靠性的标准形式。

原问题
最大化 ∏j (1 - e^{-αⱼXⱼ})(1 - e^{-βⱼLⱼ})
满足：
① ∑ Lⱼ / λᵍᵉⁿ ≤ B_T （延迟约束）
② ∑ c_tok · Lⱼ ≤ C （成本约束）
③ Lⱼ ≥ 0 （非负）
💡
目标是最优化输出 token 数量 Lⱼ 的分配。推理 token Xⱼ 视为固定参数——由模型内部策略决定。

// 问题的结构

决策变量 L_1, L_2, …, L_n（每个 Agent 的输出 token 长度）
固定参数 Xⱼ, αⱼ, βⱼ, λₜₕ, λᵍᵉⁿ, c_tok, T, C, τ̄
约束数 2 个线性的 + n 个非负
目标形式 乘积 → 凸优化可解

🔑 关键洞察
"非 LLM Agent 的可靠性 ρ_i 是固定的常数乘法因子"——
因此它们只在最终可靠性公式中贡献一个常系数，不影响 Lⱼ 的最优相对分配。

05 // log transformation

对数变换：把乘积变成求和

最大化乘积 = 最大化 log(乘积) = 最大化log 求和。这个"技巧"让问题变得可解。

对数变换后
最大化 ∑j log(1 - e^{-βⱼLⱼ})
（去掉常数项后）
满足：∑ Lⱼ ≤ B,   Lⱼ ≥ 0
🔬
为什么可以去掉常数项？
log ρ_j = log(1-e^{-αⱼXⱼ}) + log(1-e^{-βⱼLⱼ})。前一项与 Lⱼ 无关，去掉不影响最优解。
📐
凸性保证：函数 log(1-e^{-βL}) 是严格凹函数，线性约束构成凸集 → 最优解唯一且 KKT 条件完全刻画。

// 等效约束

两个约束（延迟和成本）实际上可以合并为一个有效 token 预算 B：

B = min( λᵍᵉⁿ · B_T , C / c_tok )
如果延迟更紧 λᵍᵉⁿ · B_T < C / c_tok → B = λᵍᵉⁿ · B_T
如果成本更紧 C / c_tok < λᵍᵉⁿ · B_T → B = C / c_tok

🎯 延迟和成本约束本质上是同一枚硬币的两面——都可以换算成"我们可以生成多少 token"。

05 // effective budget

有效预算：延迟 vs. 成本谁更紧？

有效预算 B = min(λᵍᵉⁿ·B_T, C/c_tok)。这是一个简单但深刻的洞察。

时间预算 B_T → 可生成 λᵍᵉⁿ·B_T 个 token
λᵍᵉⁿ·B_T = 15,000 tok
成本预算 C → 可生成 C/c_tok 个 token
C/c_tok = 12,000 tok
↓ min B = 12,000 tok（成本更紧）
⚡
瓶颈转移：优化了延迟 → B_T 变大 → 瓶颈转移到成本。反之亦然。论文的模型能告诉你哪个是当前瓶颈。

// 实际场景计算

场景谁更紧？
λᵍᵉⁿ=20tok/s, B_T=500s → 10K tok
预算 C=$30, c_tok=$0.002/tok → 15K tok		🔵 延迟更紧
λᵍᵉⁿ=50tok/s, B_T=400s → 20K tok
预算 C=$10, c_tok=$0.002/tok → 5K tok		🟢 成本更紧

💡 工程意义
如果想知道该优化延迟还是该省钱——看哪个约束决定了 B 即可。
这个框架回答了"我的努力应该放在哪"的顶层问题。

05 // water-filling theorem 🎯

注水分配定理：论文的核心贡献

这是一个漂亮的闭式解——最优分配可以直接写出来，不需要迭代求解。

Water-Filling Rule
L_j^* = [ (1/β_j) · log(1 + β_j/θ) ]_+

θ 是 Lagrange 乘子（影子价格），通过 ∑L_j^* = B 唯一确定。
[·]_+ 表示取非负部分（如果算出来负数，实际取 0）。

🔑
KKT 条件推导直觉：每个活跃 Agent 的边际收益 d(log ρ)/dL = θ 都等于同一个值

β 越小 → 分得越多（B=12,000, θ=0.000069）

💡 这就像注水——水（token）先往低处（β 小=大胃口）流
流到各处"水面"（边际收益）一样高时停止

05 // shadow price

影子价格 θ：最神奇的概念

θ 是 Lagrange 乘子，但它有非常直观的经济解释。

θ 的定义
θ = ∂/∂Lⱼ [log ρⱼ(Lⱼ^*)]
所有活跃 Agent 的边际 log-可靠性收益都等于 θ
经济含义
θ = 额外 1 个 token 带来的整体可靠性提升
θ 大 → token 非常稀缺，再增加 1 个 token 的价值很高 → 说明总预算太紧
θ 小 → token 比较充裕，增加 token 的边际价值低 → 预算相对宽裕
θ → 0 → token 无限充裕 → 每个 Agent 都可以无限提升可靠性 → 整体可靠性趋于各 Agent 的上限乘积

// 数值举例

预算 B=12,000 θ = 0.000069 最后一个 token 提升 0.0069% 的 log-可靠性
预算 B=6,000（减半） θ ≈ 0.00014 θ 翻倍→token 更稀缺
预算 B=24,000（加倍） θ ≈ 0.00003 θ 减半→token 更充裕

🎯 影子价格的工程价值
θ 是一个诊断工具：在系统中运行一段时间，统计 θ 的变化趋势——
· θ 持续变大 → 预算吃紧，该增加预算了
· θ 持续变小 → 预算充裕，可以考虑给关键 Agent 更多
· 不同 Agent 的边际收益不再等于 θ → 系统偏离了最优配置

05 // optimal reliability

最优工作流可靠性闭式解

将注水分配的结果代回原问题，得到一个优雅的最优可靠性表达式。

Corollary: Optimal Workflow Reliability
R_wf^* = ∏j (1 - e^{-αⱼXⱼ}) · βⱼ/(βⱼ+θ)
🔬
推导关键：从注水分配公式可得 e^{-βⱼLⱼ^*} = θ/(βⱼ+θ)，进而 1-e^{-βⱼLⱼ^*} = βⱼ/(βⱼ+θ)
📐
这个公式的优美之处：最优可靠性只取决于 βⱼ 和 θ，不直接依赖 Lⱼ

// 解读最优解

推理因子 1 - e^{-αⱼXⱼ} 推理做得越好，整体可靠性越高（但这里固定了）
表达因子 βⱼ/(βⱼ+θ) βⱼ 越大或 θ 越小 → 这个因子越接近 1
βⱼ 大 → 少量 token 就很高可靠 → βⱼ/(βⱼ+θ) ≈ 1 βⱼ 小 → 需要大量 token → βⱼ/(βⱼ+θ) 明显 < 1 θ 小（预算充裕）→ βⱼ/(βⱼ+θ) ≈ 1 对所有人都成立

🎯 上限：当 B → ∞ 时，θ → 0，R_wf → ∏(1-e^{-αX})——
即每个 Agent 达到其推理质量的极限

05 // intuition

为什么注水分配是最优的？

核心原理：拉平边际收益——这是经济学中的基本最优性条件。

❶
假设你有 1 个 token 还没分配 应该给谁？——给"边际收益最高"的那个 Agent。
❷
给完这 1 个之后呢？ 它的边际收益下降了（因为边际递减），下一次你可能就该给别人了。
❸
一直分到什么时候停止？ 所有 Agent 的边际收益相等时——此时再移动任何一个 token
都会降低整体可靠性（拿走一个降低的部分 > 给另一个提升的部分）

// 类比：分水喝

🧑‍🤝‍🧑 一群人从沙漠里走出来，有一壶水。给最渴的人先喝。
🧑‍🤝‍🧑 他喝了几口后没那么渴了，换给第二渴的人喝。
🧑‍🤝‍🧑 最后每个人都达到"差不多的渴的程度"→ 这就是最优分配。

⚠️ 这与直觉相反
大部分人会"均匀分配"（每人一样的 token），但这样低 β 的 Agent 还没喂饱
而高 β 的 Agent 已经吃撑了——浪费了宝贵的 token 预算。

② 差距在中等预算时最大：预算极紧时大家差不多（都没多少 token），预算极松时也都接近极限——中间区域最需要优化

07 // calibration

如何校准 β？——把理论落地

β 是 Agent 的"token 效率"参数。论文假设 β 已知，但在实际工程中需要实验校准。

1
采样数据 对每个 Agent，尝试不同的 L（如 200, 500, 1000, 2000），统计输出可靠的概率
2
拟合曲线 使用最小二乘法或最大似然估计，拟合 ρ = (1-e^{-βL}) 中的 β
3
验证 + 迭代 用注水公式分配后，再实测验证可靠性是否有提升
💡
不需要精确的 β——注水分配对 β 的误差有一定的鲁棒性。相对大小比绝对值更重要。

// 快速估算 β 的方法

单点估计法 β ≈ -ln(1-ρ_test) / L_test 在某个 L_test 下测得可靠性 ρ_test，反推 β
两点差分法 β ≈ [ln(1-ρ₁) - ln(1-ρ₂)] / (L₂ - L₁) 在两个不同 L 下测可靠性，差分得到更稳定的估计

📊 参考：我们的 waterfilling_demo.py 使用了 5 个 β 值
你可以在自己的系统上跑类似的实验来拟合 β

07 // bottleneck analysis

瓶颈识别与系统诊断

论文的框架不仅给出最优配置，还能帮助诊断系统瓶颈。

🔵 延迟瓶颈
λᵍᵉⁿ · B_T < C / c_tok
优化方向：提升 λᵍᵉⁿ（用更快的推理引擎/硬件）或 降低固定延迟 T_fixed
🟢 成本瓶颈
C / c_tok < λᵍᵉⁿ · B_T
优化方向：降低 c_tok（用更便宜的模型/API）或 增加预算
🟠 Agent 瓶颈
某个 Agent 的 β 极小，吸走大量 token
优化方向：提升该 Agent 的能力（换模型/改进 prompt/增加推理 token X）

// 诊断流程

1 测量各 Agent 的延迟参数（τ̄, λₜₕ, λᵍᵉⁿ）和成本参数（c_tok）
↓
2 校准各 Agent 的 β 值（通过采样+拟合）
↓
3 计算有效预算 B = min(λᵍᵉⁿ·B_T, C/c_tok)，确定瓶颈
↓
4 运行注水分配，得到最优 Lⱼ*
↓
5 观察 θ——如果 θ 异常大，说明预算太紧
