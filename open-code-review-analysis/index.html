// background

Code Review 的困境

👤

人力瓶颈

一个中等规模的 PR 改 2000 行代码，让团队成员逐行 review，每人至少 30 分钟。而且人的注意力有限，审到后面就容易漏。

人更擅长判断"这个设计合理吗"，不擅长"这一行有没有潜在 bug"。

🤖

AI 的承诺

LLM 可以 7×24 工作，不累不困。理论上审查质量稳定，覆盖率 100%。很多团队已经开始用 Claude Code / Codex 来做 review。

但是——真的靠谱吗？

⚠️

现实落差

实际使用中，通用 Agent 的 Review 质量经常让人失望：文件没审全、行号指东打西、换个 prompt 结果就变了。信任建立不起来。

这是 LLM 的固有问题，还是工具的架构设计问题？

阿里集团内部用两年时间回答这个问题。结论是：问题不在 LLM，在架构。

// problem · 1 / 3

痛点一：覆盖不全

场景

团队一个 PR 改了 30 个文件。你让 Claude Code "review my changes"。它从第一个文件开始审：

✅ 审了 UserService.java — 认真分析了
✅ 审了 OrderService.java — 找到了 2 个问题
⋯ ...
⚠ 审到第 10 个文件时开始简略
❌ 第 15-30 个文件只扫了一眼
❌ 有些文件根本没提

这不是"偷懒"——是 LLM 的注意力衰减：随着对话增长，模型对早期内容的关注度下降，对后续文件的审查深度自然变浅。

根因分析

纯 prompt 驱动没有硬约束。

  • • "请审查所有文件"——LLM 可能认为这是"建议"不是"命令"
  • • 没有枚举机制——LLM 只能凭记忆知道"还有哪些文件没看"
  • • context window 有限——到第 15 个文件时，前面文件的对话占了大量 token
  • • 最后一个文件只分到几十个 token 的"注意力预算"
OCR 解法：工程代码枚举所有 diff 变更 → 分配到独立的 goroutine → 每个文件从干净的 prompt 开始 → 不存在注意力衰减

// problem · 2 / 3

痛点二：位置漂移

场景

LLM 说："UserService.java 第 42 行有空指针风险"——

42: // TODO: 后续优化
43: User user = userService.findById(id);
44: if (user.getName() != null) { ← 实际在第 44 行

行号偏差 2 行看起来不多，但如果每一条 comment 都差几行，团队的人就会开始"狼来了"——再也不信任机器的结果。

为什么 LLM 数不对？

LLM 看到的 diff 是纯文本，像这样：

@@ -38,12 +38,14 @@
function foo() {
- let a = 1;
+ let a = 2;
- let b = 3;
+ let b = 4;
+ let c = 5;

LLM 要正确算出 let c = 5 在新文件的精确行号，需要：数 @@ 标记 → 记住 NewStart → 逐行加减偏移。遇到多个 hunk 穿插，谁数谁错。

OCR 解法：把 diff 解析成结构化 Hunk 对象 → NewStart + 公式计算偏移 → 100% 精确，零歧义

// problem · 3 / 3

痛点三：质量波动

场景

你的 Skill prompt 里写了"检查 SQL 注入风险"。今天审出 10 个问题，明天同样的代码只审出 3 个。发生了什么？

可能是模型版本升级了，可能 temperature 变了，可能是 prompt 中某句话的措辞稍微调整过。LLM 的本质是概率生成，同一个输入不一定产生同一个输出。

"我昨天改了一个词，今天 review 的结果跟昨天完全不一样。"
— 某团队在用通用 Agent 做 Review 两周后的反馈

三个层面的不稳定性

LLM 层面：
同一个 prompt + 同一个 input → 不同输出（概率采样）
规则层面：
"检查 SQL 注入风险" → LLM 可能理解成"检查所有 SQL"或"只检查 SELECT"
覆盖层面：
今天审了 Mapper XML，明天可能"忘了"——因为 context 里没出现相关文件
总结：
纯语言驱动 = 把"确定性"建立在"非确定性"之上
OCR 解法：规则通过 glob 匹配精确注入，是代码决定不是 LLM 自觉。文件枚举是硬逻辑。行号计算是纯算法。
三个痛点的共性：纯语言驱动的架构把"需要确定的事"和"需要推理的事"混在一起了。

// overview

Open Code Review 是什么？

阿里集团内部孵化的 AI Code Review CLI 工具。
两年内服务数万开发者、发现数百万代码缺陷。2026 年 5 月开源。

⭐ 5,388 Stars 🔧 Go 语言 📦 npm 安装 🔗 Apache-2.0

安装方式：npm i -g @alibaba-group/open-code-review
核心命令：ocr review — 读 Git diff → 调 LLM → 输出 Review Comments

一句话定位

不是"调 LLM 做 Review 的工具"，
而是一个定制的 Code Review Agent 框架——
用 Go 实现了 Agent 引擎（模板、工具、调度、压缩），
只把代码分析部分交给 LLM。

支持的模式

工作区变更 分支对比 单 Commit CI/CD 集成 自定义规则 JSON 输出 Agent 模式

CLI 命令基于 Cobra 框架，配置存于 ~/.opencodereview/config.json

// core-design

核心哲学：分界线原则

OCR 的核心理念不是"写更好的 prompt"，而是重新划定"代码做的事"和"LLM 做的事"的边界。

⚙️ 确定性工程（Go 代码）

结果 100% 可预测、可测试、可调试

  • 文件选择：枚举所有 git diff 变更 → 按 include/exclude 规则过滤 → 每个该审的文件必被派发
  • 规则匹配：doublestar 库做 glob 匹配，支持 ** 和 {java,kt} 展开。四层优先级，代码级别的决定
  • 行号解析：解析 unified diff Hunk → 从 NewStart 开始偏移计算 → 算法保障精度，不用 LLM 猜
  • 并发调度：goroutine pool + semaphore 控制并发，默认 8 路，每个文件独立 context
  • 上下文压缩：三区内存策略（冻结/压缩/活跃），60%/80% 双阈值自动触发

🧠 Agent（LLM）

动态决策，需要理解语义和上下文

  • 理解代码逻辑：分析 diff 中的上下文，判断代码是否达到了预期的行为
  • 发现缺陷：从代码 pattern 中识别 bug、安全漏洞、性能问题、边界情况
  • 撰写评语：生成可读的、有上下文支撑的 Review Comment
  • 动态查上下文：通过 file_read / code_search / file_find 等工具主动获取信息
  • 修复建议：输出 suggestion_code，不只指出问题，还给出具体改法
🎯 分界线原则
把 review 流程中 所有能用算法解决的问题剥离出来用代码解决，
只把 真正需要理解代码语义的部分 交给 LLM。

deep-dive · 管道 1/3

管道引擎：从 git diff 到并发审查

git
diff → ParseDiff
正则解析 → FilterDiffs
规则筛选 → dispatch

goroutine → Collect
统一汇总

1️⃣ ParseDiff — 解析 unified diff

用正则解析 diff --git a/... b/... 头部和 @@ ... @@ hunk 标记，输出结构化 []Diff 数组——每个元素包含 OldPath、NewPath、Diff文本、Insertions/Deletions 计数、IsNew/IsDeleted/IsBinary 标签，以及完整的 NewFileContent。

2️⃣ FilterDiffs — 规则筛选

按优先级链执行文件过滤：
二进制文件 → user exclude → 扩展名不支持 → include 逻辑 → 内置排除（test 文件等）
全部是 Go 代码的逻辑判断，不调 LLM。

3️⃣ 预过滤大文件

Diff 文本超过 80% MaxTokens 的文件会被跳过，避免撑爆 LLM context window。

4️⃣ dispatchSubtasks — 核心代码

agent.go
func (a *Agent) dispatchSubtasks(ctx) ([]LlmComment, error) {
    // goroutine pool + semaphore
    for i := range a.diffs {
        if a.diffs[i].IsDeleted { continue }
        wg.Add(1)
        sem <- struct{}{}           // 获取信号量
        go func(d model.Diff) {
            defer wg.Done()
            defer func() { <-sem }() // 释放

            // 每个文件：Plan Phase → Main Loop
            a.executeSubtask(fileCtx, d)
        }(a.diffs[i])
    }
    wg.Wait()
    return collector.Comments()
}
💡 每个文件独立 context，互不干扰。
默认 8 路并发（可调），每个文件最长 10 分钟超时。

deep-dive · 管道 2/3

Plan Phase：审查前的规划阶段

触发条件

当文件的变更行数（insertions + deletions）超过 50 行 时触发。小改动直接跳过 Plan Phase，节省时间和 token。

阈值可通过 PLAN_MODE_LINE_THRESHOLD 配置。

规划阶段做了什么

LLM 分析 diff 后输出结构化 JSON，包含：

  • • change_summary — 变更概述
  • • issues[].severity — high/medium/low
  • • issues[].tool_guidance — 需要查什么、怎么查

⚡ 注意：规划阶段的工具定义仅用于文本描述，不实际调用——本质是让 LLM "想清楚要查什么"。

规划输出示例

deep-dive · 管道 3/3

Main Loop：LLM ↔ Tools 交互循环

循环流程

1. 发送 prompt（含 diff + 规则 + 规划结果）
2. LLM 回复：文本 或 工具调用
3. 调用工具 → 结果返回 LLM
4. LLM 继续分析或提交 comment
5. 直到 task_done 或达到最大轮数
6. 下一轮或结束

🔄 默认最大 30 轮，连续 3 轮空结果会提前终止。

agent.go · performLlmCodeReview
func (a *Agent) performLlmCodeReview(
    ctx, messages, newPath,
) error {
    for toolReqCount > 0 {
        toolReqCount--

        // 发请求给 LLM（含 tools 定义）
        resp, err := llmClient.CompletionsWithCtx(ctx,
            ChatRequest{Messages: messages,
                        Tools: toolDefs})

        calls := resp.ToolCalls()

        for _, call := range calls {
            // 执行工具调用
            result := a.executeToolCall(ctx, call)

            if result.Completed { return nil }
        }

        // 追加到对话历史
        addNextMessage(assistantContent,
                      toolCalls, results, &messages)
    }
}

deep-dive · 行号 1/3

行号精度：第一层 — Hunk 匹配

LLM 输出 comment 时只有文本，没有行号。行号是 Go 代码在后台算出来的。第一层也是最核心的一层：Hunk 结构化匹配。

第一步：ParseHunks

把每文件的 diff 文本解析成 []Hunk 结构：

type Hunk struct {
  OldStart int // 旧文件起始行
  NewStart int // 新文件起始行
  Lines []HunkLine // 逐行：+/ /+(added/context/deleted)
}

正则 ^@@ -(\d+),(\d+) \+(\d+),(\d+) @@ 解析 hunk 头部。

标准化

LLM 给的是 existing_code，不一定是精确匹配。算法做三步标准化：

  • 1. strings.TrimSpace — 去首尾空格
  • 2. strings.TrimPrefix(+, -) — 去 diff 标记
  • 3. 再次 TrimSpace — 处理标记后的空格

第二步：逐行匹配 + 偏移计算

以下面这个 hunk 为例：

@@ -10,7 +10,9 @@
function foo() { ← context → 新:10
let a = 1; ← context → 新:11
- let b = 2; ← deleted → 跳过
+ let b = 3; ← added → 新:12
+ let c = 5; ← added → 新:13

OCR 的 resolveFromHunk 函数：
1. 提取 new-side 行（context + added）→ []indexedLine{lineNum, content}
2. 逐段滑动窗口匹配 existing_code
3. 匹配成功 → lineNum[0] 和 lineNum[-1] 就是精确行号范围

结果：LLM 说"有问题" → OCR 算出 "第 13 行" → 100% 精确，不靠 LLM 猜。

deep-dive · 行号 2/3

行号精度：第二、三层保障

① Hunk 匹配 匹配率约 90%——大多数 existing_code 都能在 hunk 中找到 ~90%
｜ 匹配失败时 ↓
② 全文件扫描 Hunk 匹配失败 → 读取 NewFileContent（从磁盘或 git show）→ 逐行扫描寻找连续匹配 → 匹配上的行号即为结果 Fallback
｜ 还不行时 ↓
③ LLM Re-location 前两步都失败 → 调用专用 prompt RE_LOCATION_TASK → 传入原始 diff + 匹配失败的代码片段 + 评论内容 → 让 LLM 从 diff 中提取精确代码片段 → 更新 existing_code → 重试 Hunk 匹配 最终手段

全文件扫描的原理

当 existing_code 跨 hunk 时（比如 LLM 把两段不连续的代码写在了一起导致 hunk 匹配失败），算法退而读取完整的 NewFileContent，在整个文件中滑动窗口寻找连续行匹配。

这个 fallback 能处理 Hunk 匹配无法解决的"跨段"情况。

Re-location 的专用 prompt

RE_LOCATION_TASK
系统：你是代码定位助手。给定 diff 和 review
comment，从 diff 中提取 comment 所指的精确
代码片段。

用户：Diff: {diff}
原始代码段（匹配失败）：{existing_code}
Review 评论：{suggestion_content}

规则：
1. 从 diff 中逐字复制相关行
2. 去掉 + - 前缀
3. 只输出 fenced code block

LLM 输出新 existing_code → 替换后重试 Hunk 匹配

deep-dive · 行号 3/3

为什么不直接让 LLM 报行号？

LLM 的"行号推算过程"

LLM 看到的是纯文本 diff。要算出某行在新文件中的精确行号，它要经历：

// LLM 的内部计算（概率性的）
看到 @@ -10,7 +10,12 @@ → 新文件从 10 开始
往下数：context(10) → context(11) → 删除(跳过) → 添加(12) → context(13) → 删除(跳过) → 添加(14) → 添加(15) → ...
等一下，我数到第几个 hunk 了？
这行到底是第 10 行还是第 16 行？🤔

结果：LLM 经常数错 2-5 行，尤其在有多个 hunk 或多个文件时。这就是"位置漂移"的根本原因。

OCR 的精确算法

同样是知道 @@ -10,7 +10,12 @@：

// Go 代码的计算（确定性的）
hunk := ParseHunks(diff) // 结构化解析
side := extractSideLines(hunk, newSide) // 提取 new 侧
idx := matchConsecutive(side, target) // 滑动窗口匹配
lineNum := side[idx].lineNum // 精确！

结果：行号 100% 精确。NewStart + 偏移公式，零歧义。
同样的 diff，1 万次执行，1 万次返回相同的正确行号。

核心认知：行号定位是算法问题，不是 LLM 问题。
能用代码算的就不要用 LLM 猜。

deep-dive · 上下文

三区内存压缩策略

Main Loop 中每轮对话都会累积历史。OCR 用"三区压缩"策略管理 context，保证 LLM 不断线、不丢失上下文。

❄️ 冻结区

保留前 2 条 消息不变。
（通常是 system prompt + 第一条用户消息）

保证 LLM 始终知道它的角色和任务。

📦 压缩区

中间的对话历史 → 调用另一轮 LLM 压缩成结构化摘要。

摘要包含：已发现的 issues、工具调用结论、已完成/待完成任务、当前焦点。

注入到 system prompt 的 <previous_review_summary> 标签中。

🔥 活跃区

保留最近 K 轮 完整对话。
K 的计算基于 token 预算：
budget = 80% × MaxTokens − 冻结区 − 压缩区

确保 LLM 能看到最近的上下文以继续当前分析。

⚡ 60% MaxTokens → 后台 异步压缩（不阻塞主循环，并行执行）
⚡ 80% MaxTokens → 立即 同步压缩，取消未完成的后台任务
双阈值设计：异步不阻塞，同步保安全。

deep-dive · 工具

6 个专用工具 + 异步 Comment

🔧 工具集来源

从阿里内部百万次生产调用数据中蒸馏——分析了调用频率分布、单工具重复率、新增工具对调用链的影响，最终精确到这 6 个，不多不少。

file_read 读取文件的指定行范围，支持 start_line/end_line 参数
file_find 基于 git ls-files 按文件名模糊搜索
file_read_diff 查询其他变更文件的 diff 文本（跨文件上下文）
code_search git grep 全文搜索，支持正则、大小写、文件路径过滤
code_comment 提交审查意见（含 content + existing_code + suggestion_code）
task_done 结束当前文件的审查，进入下一个文件

⚡ 异步 Comment 处理

最巧妙的设计：当 LLM 调用 code_comment 时，Go 代码不阻塞等待行号解析——把后续处理丢进 CommentWorkerPool，主循环继续跟 LLM 对话。

agent.go · executeToolCall
if t == tool.CodeComment {
    comments, _ := ParseComments(args)

    // 异步：行号解析不阻塞主循环
    pool.Submit(func() {
        resolveAndCollect(ctx, comments)
        // Hunk匹配 → 全文扫描 → Re-location
    })

    return "Comment accepted"
    // ↑ LLM 继续下一轮，后台并行算行号
}

// file_read, code_search 等同步执行
result, err := p.Execute(ctx, args)

💡 不浪费 LLM 的等待时间——它聊下一轮，后台算行号。所有文件审查完毕后统一 Await 收集。

CommentWorkerPool 用 semaphore 控制并发，默认 8 个 worker。

system_rules.json 内置 Java/Go/Rust/Kotlin/TS 等语言的默认 checklist

📋 规则格式

.opencodereview/rule.json
{
  "rules": [
    { "path": "**/*.java",
      "rule": "所有新方法必须做 null 参数校验" },
    { "path": "**/*mapper*.xml",
      "rule": "检查 SQL 注入风险" }
  ],
  "include": ["src/main/**"],
  "exclude": ["**/generated/**"]
}

glob 匹配基于 doublestar 库，支持 ** 和 {java,kt} 展开。
用 ocr rules check src/.../Foo.java 可预览命中哪条规则。

为什么比 prompt 可靠？

在 prompt 里写规则 = 建议

• LLM 可能忽略
• LLM 可能理解错
• LLM 可能只执行了一半
OCR 规则注入 = 代码决定
doublestar.Match("**/*.java", path)
→ 匹配则注入规则 → 不匹配则跳过
100% 确定，100% 一致。

// practice

三种集成方式 + 关键限制

🖥️ CLI 直接使用
ocr review --from main --to feat
ocr review --commit <sha> --preview
提交前跑一遍，适合个人开发者
🔁 CI/CD 集成
--format json 输出结构化结果，集成到 GitHub Actions / GitLab CI，每个 PR/MR 自动触发
适合团队流程，PR 自动评论
🧩 Agent Skill / 插件
支持 Claude Code、Codex、pi 等，--audience agent 模式抑制进度，支持 "review and fix"
适合写代码 → 自动审查闭环

⚠️ 关键限制

OCR 需要自己的 LLM 配置。它读自己的环境变量，直连 API：
OCR_LLM_URL + OCR_LLM_TOKEN + OCR_LLM_MODEL
不能"借用"父 Agent 的 LLM 连接。
没有独立 API Key（或用公司内部工具的 LLM），OCR 用不了。

⚖️ 对比：自写 Skill vs OCR

维度自写 SkillOCR
LLM 配置零额外，复用现有连接需独立 URL+Token
API 费用一份两份（父Agent + OCR）
行号精度LLM 估算三层算法保障
大 PR 覆盖注意力衰减并发全覆盖
结果一致性依赖 LLM 状态工程代码保障
规则执行prompt 建议glob 匹配代码决定
💡 建议：小改动自写 Skill 够用；大 PR/团队 CI 用 OCR；
没有独立 API Key 则自写 Skill 是唯一选择。但 OCR 的架构思路在任何场景都有参考价值。

// takeaways

三件事带回去

1 · 分界线原则

"能确定的写成代码，需要推理的交给 LLM。"

这个原则不只在 code review 上成立——任何 AI 工具的系统设计都可以用这条线来划。划清分界线，LLM 只做它擅长的事，工程代码做好硬约束。

2 · 行号是算法问题，不是 LLM 问题

LLM 不该数行号。Hunk 解析 + 偏移计算是纯工程逻辑，用代码保证比用 prompt 可靠一万倍。

延伸到更多场景：能用代码算的就不要用 LLM 猜。把确定性的工作从 prompt 中剥离出来。

3 · 选工具看场景和基础设施

OCR 能力很强，但需要独立的 LLM API 配置。没有独立 Key 的话，自写 Skill 是更务实的选择。

但不管用不用 OCR，它的架构设计思路——确定性的工程框架 + 有限的 Agent 决策——在任何涉及 LLM 的系统设计中都值得借鉴。
