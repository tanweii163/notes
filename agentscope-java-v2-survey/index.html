v1 Limitations · 为什么需要 v2

v1 的六大局限

当 Agent 进入生产场景后，v1 的「围绕一个 agent 生长」模式遇到了瓶颈

✗ 状态分散

Memory · Session · StatePersistence 各自维护，恢复和迁移复杂

✗ Hook 粒度粗

全局事件监听，不适合可组合的 before/after 拦截

✗ 事件不够细

粗粒度 stream 难以支撑实时 UI、HITL、Studio 观测

✗ 工具安全不足

文件 · shell 工具偏「直接执行」，缺少统一权限模型

✗ 执行环境耦合

文件系统 · 命令执行 · 工作区没有形成独立 runtime 层

✗ 生态边界不清

RAG · 长期记忆 · Session 后端容易回流到 core

Migration · 升级三条主线

v1 → v2 可以理解成三条主线

状态收敛 · 运行时迁移 · 生态扩展 — 三条主线并行，互不阻塞

Core · ReAct 执行链路详解

ReAct 循环的深入理解

ReActAgent 的核心是一个 Reasoning → Acting → Observing 的循环状态机

1

🧠 Reasoning

构建 System Prompt → Middleware.onSystemPrompt

调用 Model 流式生成 → Middleware.onReasoning → Middleware.onModelCall

输出：文本 / 思考 / 工具调用

2

🔧 Acting

收到工具调用 → PermissionEngine 判定

允许 → Middleware.onActing → Toolkit.execute

结果写入 ToolResultBlock → 保存 AgentState

3

🔄 Loop

工具结果追加到上下文 → 继续下一轮 ReAct 迭代

直到：最终回复 / HITL 暂停 / 中断 / maxIters 耗尽

保存 Session → 返回 terminal Msg

💡 关键设计：事件优先 — 文本增量 / 思考 / 工具调用 / 工具结果都发射为 AgentEvent，响应式流 — 全程 Flux/Mono 非阻塞

Core · 扩展机制迁移

Hook → Middleware 迁移示例

Hook 已标记 @Deprecated(forRemoval = true)，v2 主扩展点切到 MiddlewareBase

📉 Legacy Hook

Hook hook = new Hook() {
  @Override
  public <T extends HookEvent> Mono<T>
      onEvent(T event) {
    if (event instanceof PreReasoningEvent)
      System.out.println("推理开始");
    return Mono.just(event);
  }
};
ReActAgent.builder().hook(hook);
→

📈 v2 Middleware

MiddlewareBase mw = new MiddlewareBase() {
  Flux<AgentEvent> onReasoning(
      Agent a, ReasoningInput in,
      Function<?,Flux<AgentEvent>> next) {
    System.out.println("推理开始");
    return next.apply(input);
  }
};
ReActAgent.builder().
  middleware(mw);

Core · 主扩展机制

Middleware：比 Hook 更结构化的拦截器

v1 Hook 是事件监听式扩展 → v2 Middleware 是洋葱模型，天然支持 before/after/短路/包装

① onAgent

洋葱模型 · 整次调用计时 · 请求级 tracing · 统一异常处理

② onReasoning

洋葱模型 · 改写模型输入 · 观察推理输出 · 控制 ReAct 轮次

③ onActing

洋葱模型 · 工具调用审计 · 工具调用前后处理

④ onModelCall

洋葱模型 · 模型调用 tracing · 重试包装 · 限流统计

⑤ onSystemPrompt

管道模型 · 顺序转换 · 注入系统提示词片段 / workspace context

🎯 优点

  • 可组合的工程秩序
  • 天然支持短路/包装
  • 适合长链路系统
  • Hook 作为兼容层保留

Core · 事件流迁移

stream() → streamEvents() 迁移示例

v2 中 legacy stream(…) 已标记 @Deprecated(forRemoval = true)

📉 Legacy stream()

agent.stream(List.of(msg),
    StreamOptions.defaults())
  .subscribe(event ->
    System.out.println(
      event.getType()));
→

📈 v2 streamEvents()

agent.streamEvents(msg)
  .doOnNext(event -> {
    switch(event.getType()) {
      case TEXT_BLOCK_DELTA ->
        System.out.print(delta);
      case TOOL_CALL_START ->
        System.out.println(
          "Tool: " + name);
      case REQUIRE_USER_CONFIRM ->
        System.out.println(
          "等待确认");
      default -> {}
    }
  }).blockLast();

Core · 安全权限引擎

PermissionEngine：Agent 工具调用的安全闸门

v2 生产安全的关键能力 —— 工具执行不再只是「模型说调用就调用」

🔍 决策顺序

1 工具级 deny rules
2 工具级 ask rules
3 工具内置 checkPermissions
4 工具级 allow rules
5 BYPASS fallback
6 默认 ASK / DONT_ASK 降级 DENY

🎯 常见模式

模式行为
DEFAULT无匹配规则时 ASK 用户
ACCEPT_EDITS开发工作区，工具仍做内置检查
EXPLORE只读探索，写操作被拒绝
BYPASS默认允许（谨慎使用）
DONT_ASK无人值守时 ASK → DENY

Core · 人类确认恢复

HITL 确认恢复与 Pending Tool

当权限判定为 ASK 时，事件流产生 RequireUserConfirmEvent 并暂停执行

👤 HITL 恢复流程

1 PermissionEngine → ASK
2 发出 RequireUserConfirmEvent
3 RequestStopEvent 暂停本轮
4 后续调用需携带确认结果或 ToolResultBlock

⚙️ 配置选项

API说明
enablePendingToolRecovery(true)自动为中断的 pending tool 生成错误结果
CONFIRM_SINK_KEY (deprecated)不要依赖旧的 sink 常量

⚠️ 如果存在未完成工具调用，v2 会校验后续消息是否带有匹配的 ToolResultBlock

Migration · 三条路径

v1 → v2 迁移选择

三条主线：状态收敛 · 运行时迁移 · 生态扩展

1

📦 最小迁移

继续使用 ReActAgent，处理 Memory→Session、Hook→Middleware、stream→streamEvents

ReActAgent.builder()
  .name("assistant")
  .model(model)
  .session(new JsonSession(...))
  .sessionKey(...)
  .build();
2

🏭 生产迁移

升级到 HarnessAgent，启用工作区 · 沙箱 · 子 agent · 压缩

HarnessAgent.builder()
  .name("coding-agent")
  .workspace(Path.of("/project"))
  .compaction(config)
  .enablePlanMode()
  .build();
3

🔗 企业集成

按需引入 extensions：A2A · RAG · 长期记忆 · Redis/MySQL · Spring Boot

<dependency>
  <groupId>...</groupId>
  <artifactId>agentscope-
    extensions-session-redis</artifactId>
</dependency>

⚠️ 三条主线：状态主线 （Memory → AgentState + Session）· 运行时主线（文件/命令 → Harness）· 生态主线（RAG/记忆 → Extensions）

Architecture Review · 设计亮点

v2 五大架构优势

1 Core / Harness / Extensions 分层

轻量用户只用 core，生产用户升级 Harness，企业按需接 Extensions

2 Middleware > Hook

洋葱模型天然支持 before/after/短路/包装，比事件监听更适合长链路系统

3 AgentState 统一状态模型

context · summary · 权限 · 工具 · 任务 · 计划统一收敛，恢复点明确

4 AgentEvent 细粒度事件流

26 种事件覆盖全生命周期，支撑实时 UI · HITL · Studio · 可观测

5 PermissionEngine 生产安全

6 级决策链（deny → ask → 工具自检 → allow → BYPASS → 默认ASK），工具调用不再失控

Architecture Review · 风险与改进

主要风险 & 改进建议

⚠ 主要风险

  • Core 中 legacy 与 v2 抽象并存，新用户认知成本高
  • Core POM 出现 Harness-merged 依赖，边界需收紧
  • Builder API 功能强但入口过宽，组合语义不清晰
  • RAG/长期记忆新旧交替，统一契约尚不明确
  • AgentEvent 被外部模块依赖后需要稳定契约

★ 改进建议

  • P0：稳定 Agent / AgentEvent / AgentState / Permission 公共契约
  • P1：收紧模块边界，清理 core 不必要的后端依赖
  • P1：完善 v2 示例矩阵（streaming · permission · session · harness）
  • P2：Builder 配置对象化，减少入口膨胀
  • P2：补 Harness 子系统架构文档
