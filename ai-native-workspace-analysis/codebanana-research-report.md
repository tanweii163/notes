## 1. 产品概述

**CodeBanana** 是出门问问（Mobvoi）推出的**AI 员工协作平台**，定位为一个由 AI 驱动的协作式编程平台。它支持多位开发者在同一工作空间中实时协同工作，将实时编辑、共享环境与 AI 支持整合到一个工作流中。

### 1.1 一句话定位

> **"沟通与执行在同一个地方——决策即执行。"**

### 1.2 核心价值主张

| 维度 | 描述 |
|------|------|
| **产品形态** | AI 驱动的协作式编程平台 |
| **目标用户** | 开发者、设计师、产品经理、市场人员 |
| **核心能力** | 实时协作编程 + AI Agent 团队 + 云端开发环境 |
| **差异化** | 将沟通管理（飞书/钉钉）与执行任务（Cursor/Claude Code）合二为一 |

### 1.3 提供的能力

- 集成开发环境：具备 AI 辅助能力的全功能代码编辑器
- Git 集成：支持完整的版本控制与协作工作流
- 虚拟机工具：基于云端的开发环境
- 组织管理：支持团队协作与项目协同
- 实时沟通：与 AI Agent 及团队成员进行即时通讯
- 分析与洞察：支持性能追踪与项目分析

### 1.4 适用人群

- 希望获得更高可见性与更快迭代速度的**技术负责人和工程管理者**
- 希望更快交付产品的**创业者和小型团队**
- 希望实时了解开发进展的**产品经理**
- 需要实时协作环境进行教学、学习与代码讲解的**编程讲师和学生**

## 2. 核心理念与定位

### 2.1 为什么做 CodeBanana？

出门问问 CEO 在"超级组织"分享会中阐述了产品背景：公司里有两类工作，却一直用两套割裂的工具：

| 工具类型 | 代表产品 | 用途 |
|---------|---------|------|
| 💬 沟通管理 | 飞书、钉钉 | 开会、记录、协调 |
| ⚙️ 执行任务 | Cursor、Claude Code | 编程、设计、执行 |

**问题**：飞书里开完会、做了记录，真正执行时却在另一个工具里——发生了什么，彼此都不知道。

**CodeBanana 的解法**：沟通与任务在同一个地方，上下文实时同步，决策即执行。

### 2.2 三大核心特点

1. **🤖 Agent 为主体** — 系统核心是 Agent，人类与 Agent 协同工作，而非人类主导一切
2. **🏢 为组织而生** — 支持多人协作，权限管理明确，项目间文件系统隔离
3. **⚡ 任务与沟通合一** — 集成群聊、Agent 工作区和文件系统，沟通与任务处理无缝衔接

### 2.3 与 Cursor / Claude Code 的区别

| 维度 | Cursor / Claude Code | CodeBanana |
|------|---------------------|------------|
| **定位** | 个人开发工具 | 团队协作平台 |
| **沟通** | 无，需要切换到飞书/钉钉 | 内置群聊，沟通与任务一体 |
| **协作** | 不支持多人共享 Agent | Team Agent 支持团队共享 |
| **上下文** | 本地/单人 | 云端，团队共享上下文 |
| **组织管理** | 无 | 支持组织架构、权限管控 |
| **Skill 复用** | 无 | Skill Market，跨项目复用 |

### 2.4 核心理念

> **Agent 不只是工具，它是组织能力的延伸。CodeBanana 的目标是让每一个组织都能进化成超级组织——用 AI 倍增每一个人的产能。**

## 3. 产品双形态：Web + 桌面端

CodeBanana 有两种产品形态，共享同一套账号体系：

| 维度 | Web 版本 | CodeBanana Bot（桌面端） |
|------|---------|------------------------|
| **运行环境** | 基于浏览器 | 本地桌面应用（macOS / Windows） |
| **核心功能** | 完整产品能力 | 相同核心能力 |
| **Agent 作用范围** | 云端项目与环境 | 本地系统、文件、应用、浏览器 |
| **账号** | 共享 | 共享 |
| **项目可见性** | 云端 + 本地（在线时） | 云端 + 本地 |
| **最适合** | 协作与便捷访问 | 深度本地自动化 |

### 3.1 CodeBanana Bot 的本地 Agent 能力

- **系统控制**：读取、创建、修改、删除文件；执行终端命令与脚本
- **应用控制**：打开本地应用程序，模拟用户交互
- **浏览器自动化**：浏览网页、填写表单、提取数据
- **屏幕理解**：捕获屏幕内容、分析界面状态、校验操作结果

### 3.2 典型本地使用场景

| 场景 | 工作流 |
|------|-------|
| 文件整理 | 扫描 → 分类 → 移动文件 |
| 表单自动化 | 打开浏览器 → 填写 → 提交 |
| 报告生成 | 捕获屏幕 → 分析 → 生成报告 |
| 文件监控 | 通过 File Watch 触发工作流 |
| 应用自动化 | 打开应用 → 执行动作 |

### 3.3 Bot 专属功能

- **Bot-only Skills**：包含本地系统操作能力，仅桌面端可用
- **Cron Job 差异**：桌面端支持系统通知，File Watch 范围覆盖完整本地文件系统（Web 端仅限项目文件）
- **项目状态**：云端项目始终可用，本地项目在客户端离线时不可用

## 4. 项目体系：三位一体

CodeBanana 围绕"项目"组织所有工作。每个项目是**三位一体**的存在：

```
┌──────────────────────────────────────────┐
│              一个项目                      │
├─────────────────┬────────────────────────┤
│  🤖 一个 Agent  │ 专属 Agent，独立执行能力  │
├─────────────────┼────────────────────────┤
│  💬 一个群聊    │ 实时群聊空间，团队沟通协作  │
├─────────────────┼────────────────────────┤
│  🗂️ 一个 Workspace │ 独立文件系统 + 虚拟机  │
└─────────────────┴────────────────────────┘
```

### 4.1 项目创建方式

- **创建本地项目**：从空白项目开始
- **从 GitHub / GitLab 导入**：关联已有代码仓库
- **使用模板**：基于预配置模板（含样式、Skills 和 Agent 设置）
- **从零开始**：Custom（空项目）

### 4.2 项目列表管理

- 支持按项目类型、Owner、创建时间、最近活动时间筛选和排序
- 提供 GitHub 项目、本地项目等可视化标识
- 同一代码仓库分支的项目归类到同一个父级项目下
- 仅项目 Owner 可删除项目

### 4.3 项目与组织的绑定

每个项目都必须属于某一个特定组织，项目不能脱离组织单独存在。切换组织后，整个项目列表会随之刷新。

## 5. Agent 模式详解

每个项目群聊支持三种聊天模式，满足不同的协作需求：

### 5.1 Team Agent（团队 Agent）

**定位**：项目中的中心化 AI 协作者，团队共享

- 基于完整项目上下文运行（repo、文件、讨论记录、历史）
- 作为编码、重构、调试和任务自动化的共享执行层
- 任一时刻仅允许一名团队成员与 Agent 交互
- 默认只有项目 Owner 可以使用，其他成员需申请权限

**Agent Modes（执行模式）**：

| 模式 | 说明 |
|------|------|
| **Ask-only** | 只读模式。Agent 仅回答问题，不会修改任何文件 |
| **Coding** | 执行模式。Agent 可直接在项目中编写和编辑代码 |
| **CodeBanana (Optimized Model)** | 针对项目级任务增强优化，能力更强，可能消耗更多 tokens |
| **Claude Code (Native Model)** | 更快、更轻量，需要更明确的提示词 |

### 5.2 Private Agent（个人 Agent）

**定位**：项目内的个人 AI 助手，不影响共享工作流

- 私有的、用户级别的 Agent，独立于 Team Agent 运行
- 适用于提问、任务规划、思路探索
- 主要运行于 ask 模式（只读）
- 可与 Team Agent 会话并行使用
- 不会修改共享项目
- 使用费用计入项目 Owner

### 5.3 Discussion（人类讨论）

**定位**：纯人类沟通空间

- 共享聊天频道，用于团队讨论、对齐需求、协调工作
- 所有成员可自由参与
- 不会有 Agent 参与，不会触发代码执行或自动化

### 5.4 三模式总结

| 模式 | 谁在用 | 能否改代码 | 共享程度 | 适用场景 |
|------|-------|-----------|---------|---------|
| Team Agent | 团队（独占） | ✅ 可编码 | 共享 | 项目开发、重构、调试 |
| Private Agent | 个人 | ❌ 只读 | 私有 | 提问、规划、探索 |
| Discussion | 团队（并发） | ❌ 不涉及 | 共享 | 沟通、决策、对齐 |

## 6. 协作系统：项目与聊天

### 6.1 Chat 消息功能

聊天提供专用于人与人沟通的空间，支持：

- **消息操作**：复制、引用、转发消息
- **成员通知**：加入或退出群组时自动发送
- **搜索**：组织联系人、聊天记录、转发行为
- **消息流转**：可转发至当前群组或其他群组
- **Restore**：支持将 Agent 对话、模型记忆和项目状态回退到之前节点

### 6.2 新消息与优先级

- 新消息提示出现在 Chats、Projects、Notifications 三个关键区域
- **@mentions 优先级高于普通新消息**，帮助先处理需要立即关注的内容

### 6.3 Notification 通知系统

通知涵盖以下关键事件：
- Agent 权限申请与审批
- 组织加入申请及状态更新
- 外部项目邀请
- 账单与用量更新（达到限制时）
- 群组邀请

### 6.4 成员管理

- 项目 Owner 管理项目成员
- 支持添加组织成员（无需通知）和邀请外部联系人（发送邮件邀请）
- 可移除成员和解散群组

> 💡 特色功能：**技术支持接入** — 项目可请求专属技术支持，系统自动分配支持人员进入项目聊天

## 7. 开发工具集

开发工具通过左下角面板访问，包含四大工具：

### 7.1 Terminal（终端）

内置终端，允许开发者直接与项目环境交互：
- 安装依赖、运行脚本、调试问题
- 在项目运行环境中执行命令

### 7.2 Service（服务管理）

管理运行时进程和活动端口：
- 查看所有活跃服务与端口
- 服务启动失败时可重启 VM 环境
- 结束指定进程或端口

### 7.3 Git Status（版本控制）

提供版本控制可见性和基础操作：
- 查看当前文件变更
- 通过 Save 操作创建 commits
- 访问 commit 历史，可恢复到指定版本

### 7.4 Deploy（部署）

将项目发布为持久、可分享的链接：
- 生成稳定且不会失效的 URL
- 项目发生变更时自动更新
- 适用于静态网站和轻量级应用

## 8. 项目工作区

项目工作区是核心工作区域，提供丰富的交互能力：

### 8.1 文件导航与标签页

- 支持同时打开多个文件，以标签页形式组织
- 右键标签页可关闭当前、关闭其他、全部关闭

### 8.2 文件查看与预览

- **Markdown**：支持 Code 与 Preview 切换，预览模式可直接编辑
- **HTML / JSON / 静态文件**：支持代码视图与预览模式
- **应用项目**：支持在 Web、iOS、Android、My Device 多环境预览

### 8.3 编辑与交互

- 选中内容后可添加评论、引用到聊天、复制内容
- 评论绑定到具体行号，浮动评论面板可快速浏览和定位

### 8.4 Diff 与变更追踪

文件发生修改时，支持 Preview（当前状态）和 Diff View（对比差异）。

### 8.5 分享

某些文件类型（HTML、Markdown、JSON）支持可分享链接，无需进入完整项目即可查看。

## 9. 支持的模型

CodeBanana 支持为 Team Agent 和 Private Agent 切换不同的 LLM：

| 模型提供商 | 支持模型 |
|-----------|---------|
| **DeepSeek** | DeepSeek-V3.2, DeepSeek-R1 |
| **GLM（智谱）** | GLM-5, GLM-4.7 |
| **Kimi（月之暗面）** | kimi-k2.5, kimi-k2-thinking |
| **Doubao（豆包/字节）** | doubao-seed-2.0-pro/lite/mini/code, 1.8, 1.6 |
| **MiniMax** | MiniMax-M2.5, MiniMax-M2.1 |
| **Qwen（通义千问/阿里）** | qwen3.5-plus, qwen3-max |

> ⚠️ 由于隐私政策和监管要求，不同地区可用的模型可能有所不同。

所有模型均为**国产模型**（DeepSeek、GLM、Kimi、Doubao、MiniMax、Qwen），没有 OpenAI 等海外模型，说明 CodeBanana 主要面向中国市场，且可能致力于低成本/合规部署。

## 10. 跨项目 Agent 协作

CodeBanana 支持在同一组织内跨项目复用 Team Agent：

### 10.1 添加 Agent 流程

1. 打开 Members 面板
2. 点击 Add Agent
3. 搜索可用的 Agents（仅限同一组织下、用户是来源项目 Owner 的 Agent）
4. 通过 `@agent` 下发任务到其他项目

### 10.2 Agent 协作能力

- Agent 可以向 Discussion 发送消息
- Agent 可以通过 Contacts 在私聊中向单个用户发送消息
- Agent 可以查询通讯录成员
- Team Agent 可以读取 Discussion 中的聊天记录

## 11. 组织与团队管理

### 11.1 组织（Organization）

组织是 CodeBanana 中的核心协作单元：

| 角色 | 权限 |
|------|------|
| **Owner** | 完整控制权：管理成员、权限和计费 |
| **Admin** | 可管理组织架构与成员（可设置多个） |
| **Member** | 可访问 AI 资源并参与项目协作 |

**组织类型**：
- **Own Organization**：用户自己创建并完全控制的组织（每人只能拥有一个）
- **Work Organization**：以协作者身份加入的组织（可加入多个）

**关键机制**：用户任一时刻只能处于一个组织中，切换组织会切换整个工作空间（项目、文件、联系人）

### 11.2 部门管理

支持创建一级部门、子部门，编辑和删除部门。支持批量导入组织架构（Excel 模板）。

### 11.3 团队套餐（Team Plan）

面向需要共享资源、统一计费的团队设计：

| 特性 | 说明 |
|------|------|
| **预付费** | 每位成员每月 300 次 requests，可选 VM 配置 |
| **后付费** | 超出后按用量计费，由管理员控制 usage limits |
| **席位费** | 管理员激活时收取首个席位，后续按成员数量计费 |
| **计费周期** | 按用量达到 $2000 或每满 30 天自动扣费 |

**成员使用体验**：成员需由管理员手动添加到团队套餐，获得每月 300 次 requests 和已分配的 VM 资源。超出额度后需联系管理员提升配额。未被添加到团队套餐的成员将按个人套餐计费。

## 12. 账户与计费体系

### 12.1 账户管理

通过个人资料菜单管理所有账户选项，支持更换头像（JPG/PNG/GIF/WebP，<5MB）。

### 12.2 套餐选择

| 套餐 | Free | Personal | Professional |
|------|------|----------|-------------|
| Requests/月 | 30 | 300 | 1000 |
| VM 自定义配置 | ❌ | ✅ | ✅ |
| 年度优惠 | - | ✅ | ✅ |

### 12.3 计费核心概念

- **Request**：Agent 进行一次输出（代码生成、问答、任务执行）
- **Token**：模型实际消耗的计算量（输入 + 输出），用于按用量计费
- **计费方式**：套餐额度内从月度配额扣除，超出后切换为按用量计费

### 12.4 费用归属

- Team Agent 的使用费用由**项目 Owner** 承担
- Private Agent 的使用费用同样由**项目 Owner** 承担

## 13. 企业级能力：超级组织

### 13.1 "超级组织"理念

"超级组织"是 CodeBanana 的企业级产品哲学，核心观点：

> 公司里"沟通管理"和"执行任务"两套工具的割裂，导致决策与执行脱节。CodeBanana 将二者合一，Agent 作为组织能力的延伸。

### 13.2 系统架构

| 组件 | 说明 |
|------|------|
| 🟣 **Team Agent** | 共享 Agent，同一时间仅一人可操作，聚焦协作 |
| 🔵 **Private Agent** | 个人专属 Agent，私密空间独立运转 |
| 🟠 **Discussion** | 人类沟通区，团队讨论与决策实时同步 |
| 🟢 **Workspace** | 文件与虚拟机，三位一体协作环境的核心底座 |

### 13.3 Skill Market — 让好方法流遍全公司

- **Skill** 是每个岗位的方法论沉淀，安装即生效
- 某个人摸索出高效工作方式，打包成 Skill 发布到市场
- 其他人一键安装即可复用，组织最佳实践真正流转

### 13.4 让 Agent 主动干活（Cron Job）

- **定时触发**：设定时间或周期，Agent 自动执行任务（每日报告、定期检查）
- **事件驱动**：文件变化、消息到达、条件触发，Agent 第一时间响应
- 从被动到主动，Agent 成为真正的协作伙伴

### 13.5 人与 Agent 消息分离

Agent 工作消息多且密，与人类聊天混在一起会淹没重要信息。CodeBanana 将消息分为两个独立 Tab：

| Tab | 内容 |

### 16.2 产品定位评析

CodeBanana 的定位非常独特——它**不是**一个"AI 编程助手"，而是一个**完整的团队协作平台**，AI Agent 是其中的一等公民。它试图解决的问题不是"帮程序员写代码"，而是"让整个组织用 AI 倍增产能"。

与 Moxt 的对比思考：
- **Moxt**：更偏向**知识工作者的 AI 工作空间**（文档、调研、内容创作）
- **CodeBanana**：更偏向**开发者和工程团队的 AI 协作平台**（代码、项目、部署）
- 两者都强调"真人和 AI 同事在同一空间协作"，但 CodeBanana 更偏技术执行层面
