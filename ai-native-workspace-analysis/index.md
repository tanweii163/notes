# AI-Native 工作空间赛道分析

> 基于 Moxt 与 CodeBanana 两份调研报告的综合观察  
> 日期：2026-06-23

---

## 一、它们各自从什么起点切入

| 维度 | **Moxt** | **CodeBanana** |
|------|---------|---------------|
| **对标品类** | Notion / 飞书文档 | Cursor / Claude Code + GitHub |
| **核心场景** | 知识工作（写文档、做调研、内容创作） | 软件开发（写代码、项目管理、部署） |
| **目标用户** | 知识工作者、内容团队、个人 | 开发团队、工程管理者、创业者 |
| **AI 的定位** | 多位角色化的 AI 同事各司其职 | Team Agent + Private Agent 双模式 |
| **产出形态** | 文档、表格、Slides、Mini App、图片 | 代码项目、应用、可部署的服务 |
| **背后的公司** | 创业公司（Moxt.ai） | 出门问问（Mobvoi），AI 老牌公司 |

> **关键观察**：它们从相反的方向逼近同一个命题——Moxt 从"文档"往下走到"执行"，CodeBanana 从"代码"往上走到"沟通"。这说明"AI-Native 工作空间"这个需求不是某个细分场景的产物，而是整个工作范式正在被 AI 重塑。

---

## 二、它们共同相信什么

虽然起手式不同，但两个产品有惊人的共识：

### 共识 1：AI 不该是个聊天框

- Moxt 说：**"Moxt 不是'加了 AI 的 Notion'，也不是'换了皮的 ChatGPT'"**
- CodeBanana 说：**"沟通与任务在同一个地方，上下文实时同步，决策即执行"**

两个产品都认为，把 AI 做成一个"你问它答"的聊天框是懒惰的设计。AI 应该直接产出真实的工作成果——文件、项目、应用——而不是一段你要复制粘贴的文字。

### 共识 2：一个 AI 不够，要一支 AI 团队

- Moxt：每位 AI 同事都有自己的角色（内容编辑、销售助手、调研员……），不是一个万能助手什么都干
- CodeBanana：Team Agent（共享执行）+ Private Agent（个人探索）+ Discussion（纯人类沟通）三模式分离

它们都认为**专业化优于通用化**。一个 AI 干所有事的结果是每件事都干不好，应该像组织真人团队一样，让 AI 各司其职。

### 共识 3：AI 同事需要"成长"

- Moxt：规则 → 记忆 → 技能 三要素自动沉淀
- CodeBanana：Agent 在项目上下文中积累，Skill Market 让好的工作流在组织内传播

它们都认为 AI Agent 不该是 stateless 的。每次交互、每次纠偏，都应该让 AI 变得更懂那个团队、那个领域。用的人越多，AI 越聪明。

### 共识 4：真人和 AI 用同一套协作系统

- Moxt：同一份文件，`@真人` 和 `@AI同事` 是一样的操作
- CodeBanana：同一个项目 = 一个 Agent + 一个群聊 + 一个 Workspace

两个产品都认为，**AI 不该是一个需要你"切出去"用的工具**。它应该像同事一样，就在你工作的上下文里。

### 共识 5：平台化——技能 / 集成 / 自动化

两个产品都有：

- **技能 / 插件市场**（Moxt 的"技能"，CodeBanana 的"Skill Market"）
- **集成外部工具**（Moxt 的 Gmail/Slack/GitHub，CodeBanana 的 Git/IM Bot）
- **定时任务 / 自动化**（Moxt 的"自动化"和"Heartbeat"，CodeBanana 的"Cron Job"）

这说明它们都在往**平台化**方向走——不只做工具，要做生态。

## 三、它们各自的短板

### Moxt 的软肋

| 短板 | 说明 |
|------|------|
| **没有代码执行能力** | AI 能做 Mini App，但那是低代码/可视化层面；不支持深度编程 |
| **团队协作深度不足** | 权限体系、组织架构、企业级管理还比较浅（帮助文档大量空白） |
| **商业模式未明** | 定价模型文档为空，尚不清楚如何商业化 |
| **产品成熟度偏低** | 帮助文档 11 篇中 10 篇为空占位，团队还在打磨核心体验 |

### CodeBanana 的软肋

| 短板 | 说明 |
|------|------|
| **使用门槛高** | 面向开发者的产品，非技术用户上手有难度 |
| **模型生态局限** | 仅支持国产模型（DeepSeek/GLM/Kimi/Doubao/MiniMax/Qwen），出海受限 |
| **产品过于年轻** | 2026 年 1 月才上线第一个版本，到调研时仅 3 个多月 |
| **数据安全信任** | 作为"所有代码都在云端"的平台，企业对代码托管的信任门槛不低 |

## 四、这个赛道更大的图景

把 Moxt 和 CodeBanana 放回整个行业坐标系里看：

```text
                      知识工作（文档/内容）
                              │
                      ┌───────┴───────┐
                      │     Moxt      │
                      │  (AI-Native   │
                      │   Workspace)  │
                      └───────┬───────┘
                              │
               ┌──────────────┼──────────────┐
               │              │              │
           Notion AI     AI Native      ChatGPT
           (功能插件)   Workspace     (通用聊天)
               │         (新品类)          │
               │              │              │
               └──────────────┼──────────────┘
                              │
                      ┌───────┴───────┐
                      │  CodeBanana   │
                      │ (AI 员工协作   │
                      │  平台)        │
                      └───────┬───────┘
                              │
                      开发工作（代码/项目）
```

- **右上角 = ChatGPT 等通用聊天产品**——覆盖面最广，但深度最浅，输出止于文字
- **左上角 = Notion AI 等功能插件**——在不改变原有产品架构的前提下"加 AI"，更像是补丁而非重构
- **中间地带 = Moxt / CodeBanana 正在开拓的新品类**——以 AI 为原生公民重构工作空间

### 这个品类为什么现在出现？

三个条件在过去一年刚刚成熟：

1. **模型能力达到"可交付"阈值**——GPT-4o、Claude Sonnet 4、DeepSeek-V3 级别的模型，已能稳定产出可用的代码和文档，不再是玩具
2. **"AI Agent"范式被验证**——从 AutoGPT 到 Claude Code，从 Devin 到 Cursor，业界逐步形成了"AI 可以作为半自主 Agent 工作"的共识
3. **协作工具的 AI 化滞后**——飞书/Notion/Slack 都加了 AI 能力，但都是"插件式"补丁，没有产品从头以 AI 为第一公民来设计

## 五、几个判断

### 判断 1：这个赛道会出现多个赢家，不会一家通吃

Moxt 和 CodeBanana 选择了完全不同的切入点，但都言之有理。这个赛道会**沿着工作类型分化**：

| 方向 | 代表 | 特征 |
|------|------|------|
| 知识工作型 | Moxt | 文档驱动、内容创作、轻量协作 |
| 工程开发型 | CodeBanana | 代码驱动、项目管理、重度技术 |
| 行业垂直型 | 尚未出现成熟产品 | 销售、客服、法律、医疗等场景专属 |

就像 Notion 和 GitHub 没有互相干掉一样，AI-Native Workspace 也会分化。

### 判断 2："AI 同事的养成成本"是最大的隐性风险

两个产品都强调 AI 会"成长"——用得越多越聪明。但这里有个悖论：

> 如果一个 AI 同事需要我用 2 周才能"训练"到上手，那我还不如直接带一个真人实习生。

真人的优势：有常识、有判断力、知道什么时候该问。当前 AI 的劣势：没有常识、需要大量 explicit 纠偏、把明显错误的事做得煞有介事。

**这个赛道的产品如果能解决"低成本 onboarding"的问题——让 AI 进入 80% 工作状态的时间从几天缩短到几十分钟——那才是真正的突破。**

### 判断 3：Skill / 插件生态是决定性壁垒

两个产品都有 Skill Market 的设想。逻辑很清楚：

> 单个 AI 同事的能力天花板由模型决定，但下限由 Skill 生态决定。

一个安装了"小红书爆款文案"技能的 AI，和没装这个技能的 AI，输出质量可能是 60 分和 80 分的差别。**Skill 生态的丰富度，会成为这个赛道竞争的关键护城河。**

谁先让第三方开发者开始为它写 Skill，谁就赢了。

### 判断 4：CodeBanana 的"出门问问故事"是这个赛道最好的商业样本

CodeBanana 文档中的落地数据值得反复咀嚼：

| 指标 | 数据 |
|------|------|
| 公司人数变化 | 300 → 150，**减半** |
| 新增产品线 | +2 条 |
| 研发产能 | **翻四倍（4×）** |
| AI Token 成本 | 仅人力成本的 15% |
| 研发团队全栈比例 | 100% |
| 非研发团队全栈比例 | 30%+ |

无论这个数字是否有宣传成分，**它提出了一个无法忽视的商业命题**：

> 如果你是一家 300 人的技术公司，用 CodeBanana 可以降到 150 人，研发产出反而翻四倍。你是 CFO，你算不算这个账？

这个叙事比"提高个人效率"有力得多——它直接对着 CEO/CFO 说话。如果出门问问能持续验证这个数据，CodeBanana 的企业级销售会非常顺畅。

### 判断 5：最大的竞争对手可能还没入局

目前这个赛道的玩家都是创业公司（Moxt）或中型公司的新业务线（CodeBanana）。但想象一下：

- **Notion / 飞书**如果决定把 AI 从"插件"升级为"一等公民"
- **GitHub** 如果决定把 Copilot 从"代码补全"升级为"项目级 Agent"
- **OpenAI** 如果决定做一个自己的工作空间

这些大玩家入局时，有用户基础、有数据、有品牌信任。创业公司的窗口期可能只有 12-18 个月。

---

## 六、一句话总结

> **Moxt 和 CodeBanana 在回答同一个问题：如果 AI 不是工具，而是同事，工作空间应该长什么样？** 它们一个从文档切入、一个从代码切入，殊途同归地指向了同一个结论——未来的工作空间不是"加 AI"的现有工具，而是以 AI 为原生公民重新设计的协作系统。

这个赛道现在还很早期，Moxt 和 CodeBanana 各自都有明显的短板。但方向是对的。未来 2-3 年，我们会看到：

1. 更多玩家入场（包括大厂）
2. 产品形态向中间收敛（做文档的会加代码能力，做代码的会加文档能力）
3. 生态（Skill/插件市场）成为关键壁垒
4. "AI 同事 onboarding"的效率决定产品能否从 early adopter 走向主流

会是很精彩的一仗。
