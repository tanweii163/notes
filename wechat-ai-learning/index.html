什么是微信小程序 AI 开发模式？

这是一个 beta 阶段的智能小程序开发框架。核心思路是：开发者把小程序的功能拆成原子颗粒度的「能力片」（SKILL），微信 AI 通过 LLM 推理自动编排这些能力来响应用户对话。

一句话：微信 AI = 开发者把小程序功能拆成原子接口+原子组件（SKILL），AI 后台用 LLM 推理用户意图，动态编排调用这些能力，在对话流中即用即走，无需下载完整小程序。

核心概念

概念一句话类比
小程序 MCP微信私有的协议，让 LLM 能发现并调用小程序能力微信版的 Function Calling
原子接口最小执行单元，封装单一业务功能，在客户端 JS 环境运行一个"LLM 可调用的本地函数"
原子组件原子接口的 UI 卡片，把结构化数据渲染成 GUI对话流中的自定义组件
SKILL完整能力封装 = 说明书 + 能力声明 + 接口 + 组件一个"插件"

三方角色总览

📱 微信客户端 （用户手机）
  • 执行原子接口（JS 沙箱）
  • 渲染原子组件（glass-easel 引擎）
  • 中间件链（登录态/日志）
  • 显示对话界面 + 半屏页面
🧠 AI 后台 （腾讯云）
  • 加载 AGENTS.md / SKILL.md / mcp.json
  • LLM 推理：理解意图、选接口、填参数
  • MCP 协议：下发调用指令 + 渲染指令
  • 读 content / structuredContent 做决策
☁️ 开发者后端 （你的服务器）
  • HTTP API 提供服务
  • 按 sessionId 累积多轮状态
  • 业务数据持久化
  • LLM 从不直接访问
MCP 协议 客户端 ↔ AI 后台   |   HTTPS 客户端 ↔ 开发者后端

名字串联桥

三个文件同一个名字，LLM 和客户端各认各的：

// mcp.json（LLM 读）
{ "name": "recordSymptom", "description": "记录用户的一条症状信息", ... }

// index.js（客户端读）—— 名字必须和 mcp.json 一致
skill.registerAPI("recordSymptom", recordSymptom)

// apis/recordSymptom.js —— 函数名保持一致
async function recordSymptom({ symptom, duration, ... }) { ... }
关键：mcp.json 的 name、index.js 的 registerAPI 第一个参数、apis/*.js 的导出函数名，三者必须一致。否则 LLM 说"调 recordSymptom"，客户端找不到，直接报错。

用户说"头痛" → 客户端 → MCP → LLM 读 SKILL.md + mcp.json → 决策 → MCP 下行指令 → 客户端查 index.js → 执行 recordSymptom.js → wx.request（HTTPS） → 开发者后端

3. index.js — 注册 + 中间件（客户端执行）

const recordSymptom = require('./apis/recordSymptom')
const getDiagnosis = require('./apis/getDiagnosis')
const querySymptomDetails = require('./apis/querySymptomDetails')

const skill = wx.modelContext.createSkill('pkg/consultation-skill')

// 中间件：登录态 + 日志
skill.use(async (ctx, next) => {
  // ... 检查 sessionId，wx.login 换身份 ...
  await next()
  // ... 日志上报 ...
})

// ★ 注册接口名（必须和 mcp.json 一致）
skill.registerAPI('recordSymptom', recordSymptom)
skill.registerAPI('getDiagnosis', getDiagnosis)
skill.registerAPI('querySymptomDetails', querySymptomDetails)
中间件洋葱模型：前置登录态 → 后置日志上报 → try/catch 统一错误处理。
所有原子接口共享同一个中间件链，超时上限 300s。

4. recordSymptom.js — 原子接口实现

async function recordSymptom({ symptom, duration, severity, ... }) {
  const sessionId = wx.getStorageSync('sessionId')
  const res = await wx.request({ ← ★ 直连你的后端
    url: 'https://my-api.example.com/consultation/record',
    data: { sessionId, symptom, duration }
  })
  return {
    content: [{ type: 'text', text: `已记录症状：${symptom}` }], ← 🧠 LLM
    structuredContent: { symptom, duration, currentInfo }, ← 🧠 + 🎴
    _meta: { sessionId, recordId } ← 🎴 私有
  }
}

对话中提到的知识点

Q: 一个 app 最多挂多少个 SKILL？
30 个。官方接入指南明确写明："目前一个小程序最多可以声明 30 个 SKILL"。
Q: 一个 SKILL 最多几个原子接口和原子组件？
没有硬性数量上限，但受软约束：
• mcp.json 大小上限 24000 字节（扣除 outputSchema 和空格换行后）
• 独立分包上限 2MB
按你的 demo 一个接口约 2-3KB 来算，大约能塞 10-15 个。
Q: 什么是独立分包？
独立分包不依赖主包，可以单独下载运行。AI 模式的 SKILL 必须放在独立分包里， 这样用户说"问诊"时只需要下载问诊 SKILL 的代码（几百 KB），不用下载整个小程序（可能几 MB）。
app.json → subPackages[].independent: true 就是声明它是独立分包。
Q: 调用原子接口经过 AI 后台吗？
分两步：
① LLM 决定"调 recordSymptom" → 经过 AI 后台，MCP 下发给客户端
② 原子接口内部 wx.request() → 不经过 AI 后台，客户端直连你的后端
核心：LLM 管"调什么"，客户端管"怎么调"。LLM 从不直接访问你的后端。
Q: 微信怎么知道我有哪些原子接口？
分两层：
• AI 后台（LLM）知道 → 因为读了 mcp.json
• 客户端运行时知道 → 因为 index.js 里 registerAPI 注册了
两者通过相同的 name 串联：mcp.json 的 name = index.js 的 registerAPI 第一个参数。
Q: app.json、mcp.json、index.js 各是什么作用？
文件谁读作用
app.json → agent.skills微信编译工具"这里有一个 SKILL，代码在这个目录"
mcp.json🧠 LLM"接口叫啥、参数长啥样、绑哪个组件"
index.js📱 客户端"接口名对应哪个 JS 文件"
Q: 原子组件操作可以指定下一步调哪个接口吗？
可以。用 sendFollowUpMessage 的 api/call 类型：
viewCtx.sendFollowUpMessage({
  content: [
    { type: 'text', text: '选择拿铁' }, ← 必须传
    { type: 'api/call', data: { name: 'selectGoods', arguments: {} } }
  ]
})
如果不传 api/call，则由 LLM 自己推理下一步。
Q: 你的 demo 中的 querySymptomDetails 设计好在哪里？
把多轮对话压缩成一次交互。正常情况下，LLM 要问"持续多久？→有痰吗？→严重程度？"要 3 轮。 用了 querySymptomDetails 后，一次返回多选标签卡片 → 用户点选确认 → 一次性收集完毕。 这就是"多选标签组件"模式，比纯自然语言追问高效得多。
Q: 原子接口和原子组件为什么能共享业务状态？
它们运行在不同 JS 上下文，不共享全局变量。
状态通过两条途径传递：
① structuredContent → LLM 理解进度 → 下次调接口时把上下文当参数传入
② _meta → 直接透传给组件（LLM 不可见）
真正的持久化状态通过 sessionId 在你的后端维护。
Q: mcp.json 文件大小限制怎么计算？
"目前计算 mcp.json 的长度时，会除去所有的 outputSchema 字段及空格、换行符后再计算"。 所以实际能写的内容比 24000 字节更多，因为 outputSchema 部分不占额度。
