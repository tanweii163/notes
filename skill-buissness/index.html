复杂业务如何 Skill 化
技术预研讨论版
Research Discussion

Skill 不是业务代码本身，而是业务能力的标准化入口

以智能药师 170 为例：我们要解决的不是“把 Groovy 脚本改名为 Skill”，而是让复杂业务能力能被 agent、workflow、平台和其他团队正确发现、正确调用、正确治理。

Workflow → Agentic 厚领域后端 中等厚度 Skill Facade Policy + Eval + Trace
开场先强调讨论焦点：不是做一个新的脚本容器，而是找到复杂业务在 agentic 体系下的正确边界。智能药师是典型样本，因为它既有 workflow，也有模型、RAG、规则、推荐和卡片。

真实现状：Workflow 串节点，复杂度藏在节点里

入口SSE / completions
DUCC DAGnodes + edges
Vega 脚本约 80 个 SingletonScript
外部能力RPC / Redis / RAG / LLM
输出文本流 + 卡片 + 监控

Workflow 的优势

  • 路径确定，生产稳定性较好
  • DAG 能表达并行和 merge
  • 适合当前 170 主链路

当前痛点

  • 上下文通过 Map 和 listener 隐式传递
  • 节点内部混合业务规则、模型调用和副作用
  • 能力难复用，agent 不知道该如何正确调用
这一页说明为什么我们会纠结。workflow 并不是问题本身，问题是 workflow 节点背后的能力没有被契约化，agent 只能看到脚本名或接口名，无法理解业务边界。

核心判断：Skill 不能只是 SKILL.md，也不能只是 RPC Wrapper

Skill 的合理抽象：领域业务能力面向 agentic 调用场景的语义化、契约化、可治理 Facade。

不是

把所有复杂业务写进 Skill 脚本。

也不是

只有 Markdown 描述，或者对 RPC 做一层薄转发。

而是

定义能力语义、输入输出、边界策略、运行适配、评测和观测。

这里要把概念打准。Skill 不是替代领域服务，不是替代 workflow，也不是替代规则引擎。Skill 是给 agentic 世界看的业务能力边界。

推荐架构：厚领域后端，薄到中等厚度 Skill

Agent / Workflow / 其他团队 / 云端平台 | v Skill Facade - 语义描述 / 输入输出契约 / 调用策略 / 安全边界 / Trace | v Business Capability Registry - capabilityCode / owner / version / runtime / dependency | v Domain Capability - Java Service / Groovy / RPC / Rule / RAG / Model / Card Builder

复杂业务继续用成熟软件工程演进；Skill 负责让这些能力能被 agent 正确使用。

这页是建议的基本模型。领域业务能力和 Skill 不天然绑定，同一个领域能力可以有多个 Skill facade，同一个 Skill 也可以轻量组合多个领域能力。

Skill 至少要承载六类资产

Spec

业务语义、适用场景、不适用场景。

Contract

输入输出 schema、错误模型、版本兼容。

Policy

前置校验、输出过滤、安全边界、降级。

Adapter

对接 Vega、Java、RPC、HTTP、模型或云端 runtime。

Eval

golden case、bad case、回归测试、质量指标。

Trace

调用原因、输入摘要、输出摘要、耗时、失败原因。

这页回答“Skill 的表现力来自哪里”。不是来自 Markdown 本身，而是来自语义、契约、策略、评测和工程治理的组合。

方案光谱：不是选一种，而是按风险和成熟度组合

方案业务承载Agentic生产风险适合场景
文档型 Skill低低低业务知识沉淀、研发辅助
API Facade中中低低生产主链路、高风险能力
Policy Package中中低药品推荐、用药安全
Workflow Fragment高中中联合用药、信息采集
Atomic Tool / Mini-Agent中高高高离线预研、低风险探索
Domain Module / Marketplace高中中低长期平台化复用
不要把讨论收敛成“Skill 应该薄还是厚”。更合理的是按场景分层：生产薄一些，探索厚一些，高风险加 policy，成熟能力沉 domain module。

170 场景里的三类典型能力

01

适合原子 Skill

药品说明书检索、药品 QA RAG、药品名提取、追问生成。

02

适合 Facade + Policy

药品推荐、用药安全校验、最终回答生成。

03

适合 Fragment / Graph

联合用药、信息采集、自主服务规划、多步咨询闭环。

用 170 的真实能力说明抽象粒度差异。不是所有脚本都应该一一注册为 Skill，而是先识别领域能力，再决定 Skill 形态。

关键挑战：Skill 层太薄和太厚都有问题

太薄的问题

  • 退化成 RPC wrapper
  • agent 不理解失败原因和替代路径
  • whenToUse 只是文档，不能强执行
  • 底层领域能力不一定 agent-friendly

太厚的问题

  • 业务逻辑外溢到 Skill
  • 变成新的脚本工程
  • 测试、灰度、版本治理变复杂
  • 和 domain service 形成双轨规则

因此主路线应该是：中等厚度 Facade，复杂规则下沉，调用策略可执行。

这一页回应“薄前台厚后台”这个想法的挑战。我们认可方向，但要避免薄成无治理转发，也避免厚成业务泥潭。

建议演进路径

短期：识别能力，不动主链路

  • 170 Skill Catalog
  • 脚本到能力映射
  • 上下文 key 字典
  • Skill Contract 模板

中期：Adapter + Policy + Eval

  • SkillExecutor + VegaSkillAdapter
  • required context checker
  • safety policy
  • golden cases / bad cases

预研：低风险 Agent Tool

  • 说明书摘要
  • QA 检索
  • 追问生成
  • 推荐理由生成

长期：Domain Module + 平台

  • 高复用能力下沉 Java service
  • 本地 skill package
  • 云端 registry / marketplace
  • 权限、版本、SLA、评测分
这里给出可执行路线。第一步不是重构，而是能力地图；第二步是 adapter 和 trace；第三步才是 agentic POC；长期再做平台化。

一个示例：药品推荐 Skill 不应该长成脚本名

不推荐

skillCode: RobotChatbotDrugRecommendSlotComponentFlowNode_Vega input: Map<String,Object> output: Map<String,Object>

推荐

skillCode: pharmacist.drug-recommend input: userQuestion patientProfile drugContext conversationContext output: status recommendedDrugs recommendReason safetyWarnings noRecommendReason runtime: type: vega script: RobotChatbotDrugRecommendSlotComponentFlowNode_Vega
重点是 skillCode 必须是业务语义，runtime script 只是实现细节。否则 registry 只是脚本目录，不是能力目录。

最终建议

Skill 的价值不是让复杂业务少写工程代码，而是让复杂业务能力能被 agentic 系统正确发现、正确调用、正确治理、正确复用。

生产链路

Contract + Facade + Policy + Eval + Trace。

预研探索

低风险能力允许 Atomic Tool / Mini-Agent。

长期平台

Domain Module + Registry + Marketplace。

对智能药师 170：先做能力地图和 Vega Adapter POC，再逐步把高复用能力下沉为领域模块。

收束观点：不要把业务迁移到 skill 脚本。Skill 是连接成熟领域工程和 agentic 调用的标准化边界。
